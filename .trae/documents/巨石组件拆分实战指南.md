## 2. 拆分步骤（按规范执行）

### 步骤 1：触发评估

对照规则 [§1 触发条件](../rules/巨石组件拆分规范.md#1-何时拆分触发条件三选一即触发)：

- [x] 行数 ≥ 800 行（1130+ 行 ✓）
- [x] Composable ≥ 5 个（5 个 ✓）
- [x] 职责 ≥ 3 个（6 个 ✓）

**结论**：三项全中，触发拆分。

### 步骤 2：询问用户

按规则 [调用 AskUserQuestion 工具询问.md](../rules/调用AskUserQuestion%20工具询问.md)，**必须**先向用户确认是否拆分。

向用户呈现：
- 当前文件状态（行数 / 职责 / 符号数）
- 拆分方案（子包结构 + 薄壳兼容）
- 期望收益（PR 冲突率、文件行数）
- 风险评估（MainScreen 0 改动的可行性）

### 步骤 3：设计子包结构

按规则 [§2 拆分结构](../rules/巨石组件拆分规范.md#2-拆分结构按职责切分)，采用 `model / sections / dialogs` 三层：

| 层 | 职责 | AppDrawer 案例 |
| --- | --- | --- |
| **model** | 纯数据/操作类型 | CategoryAction、DateTypeAction |
| **sections** | 主分区 UI | UserHeader、CategoryGroup、InspirationFilter、DateTypeFilter、ProfileQuickNav |
| **dialogs** | 弹窗和操作菜单 | AddCategory、RenameCategory、DeleteCategoryConfirm、Category/DateType OperationSheet |

### 步骤 4：识别可见性调整

按规则 [§4 可见性调整](../rules/巨石组件拆分规范.md#4-可见性调整)，列出所有需 `private → internal` 的符号：

| 原位置 | 原可见性 | 新位置 | 新可见性 | 原因 |
| --- | --- | --- | --- | --- |
| `CategoryItem` | `private` | `CategoryItemRow.kt` | `internal` | 5 个 section 复用 |
| `DRAWER_ICON_ALL` | `private const` | `CategoryItemRow.kt` | `internal const` | 多 section 共用 |
| `DRAWER_ICON_UNCATEGORIZED` | `private const` | `CategoryItemRow.kt` | `internal const` | 多 section 共用 |
| `categoryIcons` | `private` | `CategoryItemRow.kt` | `internal` | 多 section 共用 |

### 步骤 5：识别 FQN 调用点

按规则 [§3 薄壳层规则](../rules/巨石组件拆分规范.md#3-薄壳层facade规则)，薄壳与子包同名函数需用 FQN：

| 薄壳函数 | 子包函数 | 冲突原因 |
| --- | --- | --- |
| `AddCategoryDialog` | `appdrawer.dialogs.AddCategoryDialog` | 同名 |
| `RenameCategoryDialog` | `appdrawer.dialogs.RenameCategoryDialog` | 同名 |
| `DeleteCategoryConfirmDialog` | `appdrawer.dialogs.DeleteCategoryConfirmDialog` | 同名 |
| `CategoryOperationSheet` | `appdrawer.dialogs.CategoryOperationSheet` | 同名 |
| `DateTypeOperationSheet` | `appdrawer.dialogs.DateTypeOperationSheet` | 同名 |

**处理方式**：薄壳 `AppDrawer.kt` 内部调用时用 FQN（如 `com.corgimemo.app.ui.components.appdrawer.dialogs.AddCategoryDialog(...)`）。

### 步骤 6：识别 sealed class 兼容点

按规则 [§3 薄壳层规则](../rules/巨石组件拆分规范.md#3-薄壳层facade规则)，sealed class 用 typealias 兼容：

```kotlin
// 薄壳: AppDrawer.kt
import com.corgimemo.app.ui.components.appdrawer.model.CategoryAction as CategoryActionImpl
import com.corgimemo.app.ui.components.appdrawer.model.DateTypeAction as DateTypeActionImpl

// Kotlin typealias 编译期展开为原类型，MainScreen 的
// `is CategoryAction.ShowMenu` pattern match 完全等价。
typealias CategoryAction = CategoryActionImpl
typealias DateTypeAction = DateTypeActionImpl
```

### 步骤 7：识别 Composable 默认值陷阱

按规则 [§7 常见陷阱](../rules/巨石组件拆分规范.md#7-常见陷阱)，识别不能给默认值的参数：

| 参数 | 不能给默认值的原因 | 解决方案 |
| --- | --- | --- |
| `sheetState: SheetState` | `rememberModalBottomSheetState()` 是 Composable 函数 | 调用方在 Composable 作用域创建并传入 |
| `sheetState: SheetState`（CategoryOperationSheet） | 同上 | 同上 |
| `sheetState: SheetState`（DateTypeOperationSheet） | 同上 | 同上 |

### 步骤 8：创建子包文件

按设计的 12 个子文件逐一创建，每个文件保持单一职责。**关键原则**：
- model 文件只含 sealed class（无 UI 依赖）
- sections 文件每个只含一个分区相关 Composable
- dialogs 文件每个只含一个弹窗（OperationSheets.kt 例外，合并 2 个相似菜单）

### 步骤 9：精简原文件为薄壳

按规则 [§3 薄壳样板](../rules/巨石组件拆分规范.md#3-薄壳层facade规则)，原文件精简为：
- 1 个 typealias 区（sealed class 兼容）
- 1 个主入口转发（`AppDrawerContent` 透传 24+ 参数）
- 5 个弹窗转发（用 FQN）
- 文件头注释（说明拆分原因和外部使用约定）

**关键约束**：薄壳 < 150 行，不含业务逻辑。

### 步骤 10：检查 import 完整性

按规则 [import 语句检查.md](../rules/import语句检查.md)，逐一检查 12 个新文件的 import：
- 是否有未使用的 import
- 是否有遗漏的 import（特别是 FQN 调用未 import 的场景）
- 是否有循环 import（同包内）

### 步骤 11：薄壳约束检查

薄壳层完成后对照检查：

- [x] 文件行数 < 150 行（实际 129 行 ✓）
- [x] 0 业务逻辑（仅参数透传 ✓）
- [x] 0 Composable 内部状态（无 `remember` / `mutableStateOf` ✓）
- [x] sealed class 用 typealias 兼容 ✓
- [x] FQN 解决同名冲突 ✓
- [x] sheetState 不给默认值 ✓

## 6. 经验沉淀（踩坑记录）

### 6.1 拆分前容易忽略的细节

| 细节 | 教训 |
| --- | --- |
| **Kotlin typealias 对 sealed class 有效** | 编译期展开，pattern match 完全等价，零运行时成本 |
| **Composable 函数不能作为默认值** | `rememberModalBottomSheetState()` 必须在 Composable 作用域内调用 |
| **private const 跨文件不可见** | 即使在同包，不同文件也是不同作用域，必须 internal |
| **同名函数即使在子包也冲突** | Kotlin import 不区分路径层级，薄壳必须用 FQN |

### 6.2 拆分中的判断点

| 判断点 | 决策 |
| --- | --- |
| OperationSheets 是否合并到一个文件 | 合并（CategoryOperationSheet + DateTypeOperationSheet 都是 BottomSheet，结构相似） |
| CategoryItemRow 独立文件 vs 内联 | 独立文件（5 个 section 共用，独立后修改影响面更小） |
| DrawerUserHeader 独立 vs 内联 | 内联到 AppDrawerContentImpl（只在一个地方用，无需独立） |
| 是否删除原文件 | **保留薄壳**（MainScreen 已 import，删除会全报错） |

### 6.3 拆分后验证的优先级

1. **编译通过**（最高优先级，但 AI 不擅自执行）
2. **薄壳 < 150 行**（物理约束）
3. **MainScreen 0 改动**（兼容性约束）
4. **所有功能正常**（行为约束）
5. **sealed class pattern match 正常**（类型约束）
