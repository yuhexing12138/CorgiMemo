# 创建待办时柯基互动方案

## 一、问题分析

### 1.1 当前实现

| 场景          | 当前行为                                           | 问题             |
| ----------- | ---------------------------------------------- | -------------- |
| 点击 "+" 创建待办 | 跳转到 `TodoEditScreen` 全屏页面                      | 用户看不到柯基姿态变化    |
| 创建待办页面加载    | 调用 `homeViewModel.setPoseForCreating()` 设置坐立姿态 | 用户看不到这个变化      |
| 保存待办后返回     | 调用 `homeViewModel.setPoseForLoading()` 设置奔跑姿态  | 用户看不到这个变化      |
| 返回首页        | 柯基恢复默认姿态                                       | 用户看到，但之前的变化被忽略 |

### 1.2 用户期望

**目标**：让用户能够和柯基形成互动效果，即使在创建/编辑待办时也能看到柯基的反应。

## 二、解决方案对比

### 方案对比表

| 方案                   | 描述                                 | 优点             | 缺点                       | 推荐度   |
| -------------------- | ---------------------------------- | -------------- | ------------------------ | ----- |
| **方案 A：底部弹窗创建待办**    | 不跳转到新页面，使用 `BottomSheet` 在底部弹出创建面板 | 柯基始终可见，用户体验流畅  | 需要重构 TodoEditScreen，改动较大 | ⭐⭐⭐⭐⭐ |
| **方案 B：半透明页面**       | TodoEditScreen 使用半透明背景，可以看到下方的柯基   | 改动较小，实现简单      | 半透明可能影响输入体验，需要调整布局       | ⭐⭐⭐⭐  |
| **方案 C：创建待办页面添加小柯基** | 在 TodoEditScreen 顶部添加一个小型柯基组件      | 用户在创建待办时也能看到柯基 | 柯基会出现两次，体验可能不够统一         | ⭐⭐⭐   |
| **方案 D：过渡动画**        | 页面跳转前先展示柯基姿态变化动画，再跳转               | 过渡效果好，体验流畅     | 只能看到跳转时的动画，创建过程中看不到      | ⭐⭐    |

### 2.1 方案详解

#### 方案 A：底部弹窗创建待办（推荐）

**实现方式**：

* 使用 `ModalBottomSheet` 或 `BottomSheetScaffold`

* 点击 "+" 时，从底部滑出创建待办面板

* 柯基始终在背景可见

**效果示意**：

```
┌─────────────────────────────┐
│  ┌───────────────┐          │
│  │     🐕🐾      │ ← 柯基可见│
│  │   坐立姿态     │          │
│  └───────────────┘          │
│                             │
│  ═════════════════════════  │
│  ┌───────────────────────┐  │
│  │  [标题输入框]          │  │
│  │  [详情输入框]          │  │ ← 底部创建面板
│  │  [优先级][日期]        │  │
│  │  [保存按钮]            │  │
│  └───────────────────────┘  │
└─────────────────────────────┘
```

**交互流程**：

1. 点击 "+" → 柯基切换为坐立姿态（认真听）
2. 底部面板滑出 → 柯基始终可见
3. 用户输入时 → 柯基保持坐立姿态
4. 点击保存 → 柯基切换为庆祝姿态
5. 面板关闭 → 柯基恢复趴卧姿态

**需要修改的文件**：

* `HomeScreen.kt` - 添加 BottomSheet 容器

* 新建 `TodoCreateBottomSheet.kt` - 提取创建表单

**优点**：

* 柯基全程可见，互动体验完整

* 符合现代移动端设计风格

* 用户反馈直接可见

**缺点**：

* 需要重构部分代码

* 编辑待办场景需要额外考虑

***

**效果示意**：

**需要修改的文件**：

**优点**：

#### 方案 C：创建待办页面添加小柯基

**实现方式**：

* 在 TodoEditScreen 顶部添加一个小型柯基组件

* 姿态与首页柯基同步

**需要修改的文件**：

**优点**：

* 用户在创建待办时能看到柯基

**缺点**：

* 柯基会出现两次（首页和编辑页）

* 体验不够统一

## 三、推荐方案：方案 A + 编辑页面保持不变

### 3.1 推荐理由

1. **用户体验最佳**：柯基全程可见，互动体验完整
2. **符合设计规范**：底部弹窗是移动端常见交互模式
3. **可扩展性强**：未来可以添加更多互动效果
4. **改动可控**：创建待办使用弹窗，编辑待办保持原页面

### 3.2 交互设计

| 操作        | 柯基反应         | 用户体验            |
| --------- | ------------ | --------------- |
| 点击 "+" 创建 | 坐立姿态 🐕（认真听） | 感受到柯基在"认真听"用户输入 |
| 输入过程中     | 坐立姿态（偶尔眨眼）   | 柯基保持"等待"状态      |
| 点击保存      | 庆祝姿态 + 动画 🎉 | 感受到柯基"为用户高兴"    |
| 保存完成      | 趴卧姿态恢复       | 柯基"休息"等待下次任务    |

### 3.3 技术实现

#### 步骤 1：提取创建待办表单组件

* 创建独立的 `TodoCreateForm` Composable

* 包含标题、详情、优先级、日期、保存按钮

* 接收保存回调

#### 步骤 2：在 HomeScreen 中添加 BottomSheet

* 使用 `ModalBottomSheet` 或 `BottomSheetScaffold`

* 柯基区域在背景，底部面板在前景

* 打开/关闭时同步柯基姿态

#### 步骤 3：调整导航逻辑

* 点击 "+" 不再导航到 TodoEditScreen

* 而是打开 BottomSheet

* 编辑待办仍然导航到 TodoEditScreen（保持不变）

#### 步骤 4：增强交互反馈

* 输入时柯基偶尔眨眼

* 保存成功时播放庆祝动画

* 关闭面板时柯基恢复默认姿态

### 3.4 文件修改清单

| 文件                            | 修改类型 | 说明                            |
| ----------------------------- | ---- | ----------------------------- |
| `HomeScreen.kt`               | 增强   | 添加 BottomSheet 容器，调整 "+" 按钮行为 |
| 新建 `TodoCreateBottomSheet.kt` | 新增   | 创建待办的底部弹窗表单                   |
| `HomeViewModel.kt`            | 增强   | 添加表单状态管理方法                    |
| `TodoEditScreen.kt`           | 保持   | 编辑待办页面保持不变                    |

### 3.5 关键代码设计

#### BottomSheet 状态管理

```kotlin
val bottomSheetState = rememberModalBottomSheetState()
var showCreateTodo by remember { mutableStateOf(false) }

// 点击 "+" 时

homeViewModel.resetPoseToDefault()  // 恢复趴卧
```

## 五、风险与注意事项

### 5.1 潜在风险

| 风险   | 说明                    | 缓解措施                             |
| ---- | --------------------- | -------------------------------- |
| 布局冲突 | BottomSheet 可能与现有布局冲突 | 使用 ModalBottomSheet，独立于 Scaffold |
| 状态同步 | 表单状态与 ViewModel 同步问题  | 使用单独的 ViewModel 或传递回调            |
| 用户习惯 | 用户可能习惯了全屏编辑           | 在首次使用时提示或允许切换模式                  |

### 5.2 注意事项

1. **保持向后兼容**：编辑待办页面保持不变
2. **动画流畅**：BottomSheet 动画要平滑
3. **响应式布局**：支持不同屏幕尺寸
4. **键盘适配**：弹出键盘时调整布局
