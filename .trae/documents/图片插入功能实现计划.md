```

❌ 无图片压缩（直接存储原始 Uri）
```

## 🎯 实现方案设计

### 方案对比：独立表 vs JSON 字段

| 维度        | 方案A: card\_image 独立表 | 方案B: 现有 JSON 字段（推荐） |
| --------- | -------------------- | ------------------- |
| **数据一致性** | ✅ 强一致性，外键约束          | ⚠️ 应用层保证            |
| **查询性能**  | ✅ 可建索引，JOIN查询        | ✅ 直接读取，无需JOIN       |
| **实现复杂度** | ❌ 高（新表+DAO+迁移+关联）    | ✅ 低（复用现有字段）         |
| **兼容性**   | ❌ 需要数据库迁移 v18→v19    | ✅ 待办只需 v18→v19 加字段  |
| **代码改动量** | ❌ 大（新建多个文件）          | ✅ 小（增强现有组件）         |
| **适用场景**  | 图片需独立管理/复用/统计        | 图片依附于卡片，随卡片删除       |

**推荐方案B理由**：

1. 图片生命周期与卡片绑定（卡片删除时图片应一并清理）
2. 现有 `inspirations` 和 `special_dates` 已使用此模式
3. 保持三个功能的数据模型一致性
4. 减少数据库迁移风险和复杂度

### 第三阶段：权限处理与错误处理

#### 3.1 相机权限处理

**权限列表**:

* Android 13+: `android.permission.CAMERA`

* 所有版本: 需要在 `AndroidManifest.xml` 声明

**实现位置**: `ImagePickerDialog.kt`

* 使用 `ActivityResultContracts.TakePicture()`

* 权限请求前检查 `Manifest.permission.CAMERA`

* 被拒绝时引导用户去设置

#### 3.2 存储权限说明

* **相册选择**: 使用 `PickVisualMedia` API，不需要存储权限（Android 4.4+）

* **相机拍照**: 照片输出到应用内部存储，不需要存储权限

* **结论**: ✅ 无需 `READ/WRITE_EXTERNAL_STORAGE` 权限

#### 3.3 错误处理机制

| 场景     | 处理方式                         |
| ------ | ---------------------------- |
| 图片压缩失败 | 显示 Toast "图片处理失败，请重试"，不添加到列表 |
| 存储空间不足 | 显示 Toast "存储空间不足"            |
| 文件已删除  | 缩略图显示占位符，点击时提示"图片不存在"        |
| 权限被拒绝  | 显示说明对话框，引导去设置                |

## 🔍 关键技术决策

### 1. 为什么不用独立的 card\_image 表？

* **简洁性原则**: 图片与卡片生命周期一致，无需独立管理

* **一致性原则**: 灵感和日期已使用 JSON 字段，待办应保持统一

* **性能考虑**: 避免 JOIN 查询，读取更快

* **维护成本**: 减少数据库迁移复杂度

### 2. 为什么选择 Coil 而不是 Glide？

* Compose 原生支持（`AsyncImage`）

* 轻量级（比 Glide 小 50%+）

* Kotlin 协程原生集成

* 社区活跃，Compose Multiplatform 支持

### 3. 图片压缩策略为什么选 2048px @ 85%？

* **2048px**: 适合现代手机屏幕（1080p-4K），保证清晰度

* **85% JPEG 质量**: 视觉无损，文件大小减少 60-80%

* **参考**: 微信朋友圈、Instagram 均采用类似参数

### 4. 为什么使用内部存储而不是外部存储？

* **隐私性**: 图片随应用卸载自动清理，无残留

* **安全性**: 其他应用无法访问

* **无需权限**: 不需要申请存储权限

* **可靠性**: 不会被用户或清理工具误删

## ⚠️ 注意事项与风险

### 兼容性风险

* **Android 13+ 分区存储**: 必须使用内部存储或 MediaStore API

* **相机 Intent 兼容性**: 不同 OEM 定制系统行为可能不同

* **Coil 版本**: 确保 Compose 版本兼容（建议 Coil 2.x）

### 性能风险

* **大图片 OOM**: 必须先采样再解码（inSampleSize）

* **大量图片卡顿**: LazyRow + Coil 缓存可缓解

* **压缩耗时**: 应在后台线程执行（Dispatchers.IO）

### 用户体验风险

* **权限弹窗疲劳**: 首次使用时清晰说明为何需要权限

* **存储空间监控**: 压缩前检查可用空间

* **失败重试机制**: 网络异常或存储异常时的友好提示
