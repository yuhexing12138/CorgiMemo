## Context（为什么要做）

用户提出三个与"关联"功能相关的建议，都是 CorgiMemo 已上线功能在真实数据量下暴露的问题：

1. **sourceId=0 脏数据**：用户在"新建模式"下（卡片尚未保存到数据库）打开"关联选择器"添加关联时，`InspirationEditViewModel.addRelation`（`viewmodel/InspirationEditViewModel.kt:2098`）和 `SpecialDateViewModel.addRelation`（`viewmodel/SpecialDateViewModel.kt:731`）都会用 `existingInspiration?.id ?: 0L` / `_editingDate.value?.id ?: 0L` 计算 sourceId，**直接把** **`sourceId=0`** **写入** **`card_relations`** **表**；同时 `CardRelationRepository.addRelation` 还会自动插入反向 B→A 关联（其中 `targetId=0`）。这导致数据库留下指向"卡片 0"的脏记录，新建保存后这些关联仍然指向不存在的卡片。`TodoEditViewModel.addRelation`（L2585）已经在 `todoId == 0L` 时显式拒绝（"请先保存待办再添加关联"），是反例参考。

2. **searchCards 性能问题**：`CardRelationRepository.searchCards`（`data/repository/CardRelationRepository.kt:218`）对 inspirations 和 special\_dates 用 `getAllInspirationsBlocking()` + in-memory filter，**全量扫描后再过滤**。当数据量上升（每个用户灵感数 > 500）时，关联选择器首次打开会卡顿 200ms+。事实上三个 DAO 都已有 LIKE 模糊查询方法（`searchTodos` / `searchInspirations` Flow 版 / `searchSpecialDates` Flow 版），只是 Repository 没有用。

3. **🔗N 不可点击**：`RelationCountBadge`（`ui/components/TodoListItem.kt:1305`，private）只在卡片底栏展示"🔗×3"纯文字徽章，没有任何点击反馈。HomeScreen / InspirationListItem 上每天 50+ 用户查看关联数时被吐槽"看到数字但跳不进去"。

**预期结果**：

* 修掉脏数据生成路径 + 自动修复已有脏数据

* 关联搜索 P95 延迟 < 50ms（中等数据量 1000 条内）

* 首页卡片上点 🔗N 直接弹关联列表 BottomSheet，可逐项进入关联卡片详情

## 实施计划

### 改动 1：sourceId=0 自动迁移（核心修复）

#### 1.1 新增 Repository 方法

`data/repository/CardRelationRepository.kt`：新增 `fixupZeroSourceRelations(newSourceType, newSourceId)`：

* 在 Room 事务（`@Transaction` 标注的 DAO 方法调用）内做两个 UPDATE：

  * **正向 A→B**：`UPDATE card_relations SET sourceId = :newSourceId WHERE sourceType = :newSourceType AND sourceId = 0`

  * **反向 B→A**：`UPDATE card_relations SET targetId = :newSourceId WHERE targetType = :newSourceType AND targetId = 0`

* 函数注释明确：仅在新建保存成功后调用一次，处理"新建模式下提前 addRelation"的脏数据。

* 不需要新增 DAO 方法，复用已有的 `@Query` 加一个 UPDATE 语句即可，或在 `CardRelationDao` 加一个 `@Transaction suspend fun fixupZeroSourceRelations(...)`（更安全）。

**新增 DAO 方法**（`data/local/db/CardRelationDao.kt`）：

```kotlin
@Query("UPDATE card_relations SET sourceId = :newSourceId WHERE sourceType = :newSourceType AND sourceId = 0")
suspend fun fixupZeroSourceId(newSourceType: String, newSourceId: Long)

@Query("UPDATE card_relations SET targetId = :newSourceId WHERE targetType = :newSourceType AND targetId = 0")
suspend fun fixupZeroTargetId(newSourceType: String, newSourceId: Long)
```

**Repository 包装**：

```kotlin
suspend fun fixupZeroSourceRelations(newSourceType: String, newSourceId: Long) {
    cardRelationDao.fixupZeroSourceId(newSourceType, newSourceId)
    cardRelationDao.fixupZeroTargetId(newSourceType, newSourceId)
}
```

> **非事务的理由**：两个 UPDATE 独立幂等，重复运行安全；不引入事务避免与已有 `@Transaction` 注解冲突。`fixupZero*` 写入在性能上无敏感路径（仅新建保存末尾一次）。

#### 1.2 三个 ViewModel 接入保存末尾

| 文件                                      | 位置                     | 改动                                                                                                                                                                            |
| --------------------------------------- | ---------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `viewmodel/InspirationEditViewModel.kt` | `performSave()` L850 后 | 新建分支 `inspirationRepository.insert(inspiration)` 返回 `inspirationId` 后，调用 `cardRelationRepository.fixupZeroSourceRelations("inspiration", inspirationId)`                      |
| `viewmodel/SpecialDateViewModel.kt`     | `saveDate()` L370 后    | `repository.insert(newDate)` 返回 `newId` 后，调用 `cardRelationRepository.fixupZeroSourceRelations("date", newId)`                                                                 |
| `viewmodel/TodoEditViewModel.kt`        | `performSave()` L947 后 | `todoRepository.insertTodo(todo)` 返回 `todoId` 后，调用 `cardRelationRepository.fixupZeroSourceRelations("todo", todoId)`。**注**：Todo 路径理论上不需要（addRelation 已保护），但作为防御性修复保留，避免历史脏数据。 |

所有三处插入位置都在 `if (existingXxx == null) { ... }` 新建分支内末尾，确保只在新建模式下执行。

#### 2.2 Repository 重构

#### 2.4 索引补强（Migration v44 → v45）

`data/local/db/CorgiMemoDatabase.kt`：

新增 `MIGRATION_44_TO_45`，目标是为三表 title 字段加索引（**注：SQLite LIKE '%x%' 实际无法走 B-Tree 索引**，但能加速 ORDER BY createdAt 与未来 FTS5 切换）：

```sql

CREATE INDEX IF NOT EXISTS index_inspirations_title ON inspirations(title);
CREATE INDEX IF NOT EXISTS index_special_dates_title ON special_dates(title);
```

**Entity 同步**：在 `TodoItem.kt:10` / `Inspiration.kt:14` / `SpecialDate.kt:14` 的 `indices` 数组里同步追加 `Index(value = ["title"])`。依据 `.trae/rules/entity与 migration同步检查.md` 规则保持 SQL DEFAULT / 索引名一致。

**版本号**：database version 44 → 45，`addMigrations(...)` 列表追加 `MIGRATION_44_TO_45`。
