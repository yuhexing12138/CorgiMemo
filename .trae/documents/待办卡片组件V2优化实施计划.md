## 📋 项目概述

### 目标
在 V1.0（排序弹窗、操作弹窗、编辑页增强）的基础上，继续深度优化待办卡片组件，涵盖以下三大优先级的 9 个功能点：

### 设计原则
- **增量增强**: 在现有代码基础上添加新功能
- **高内聚低耦合**: 每个功能独立成模块，可单独测试和维护
- **用户体验优先**: 所有交互符合 Material Design 3 规范
- **性能优化**: 动画使用 Compose Animation API，避免过度绘制

#### A.1 富文本编辑器集成 (复杂度: 🔴 高)

**目标**: 为 TodoEditScreen 提供真正的富文本编辑能力

**技术方案**:
```

├── RichTextEditor.kt          ← 新建：核心富文本编辑组件
│   ├── AnnotatedString 状态管理
│   ├── 工具栏按钮状态（加粗/斜体等）
│   ├── SpanStyle 应用逻辑
│   └── 光标位置追踪
├── TextFormatToolbar.kt        ← 新建：格式化工具栏子组件
│   ├── 字体样式组（B/I/U/S）
│   ├── 列表格式组（•/1./☑️）
│   ├── 对齐方式组（左/中/右）
│   └── 颜色/图片插入
└── MarkdownParser.kt           ← 新建：Markdown 导入导出
    ├── 解析 Markdown → AnnotatedString
    └── 导出 AnnotatedString → Markdown
```

**接口定义**:
```kotlin

    isBold: Boolean,
```

## 🔍 技术决策与权衡

### 决策 1: 富文本编辑器实现方式

| 方案 | 优点 | 缺点 | 推荐 |
|------|------|------|------|
| **A: AnnotatedString + TextField** | 原生 Compose、轻量、易维护 | 功能相对基础 | ⭐⭐⭐⭐⭐ |
| B: WebView + 富文本 JS 库 | 功能强大 | 重量大、性能差 | ⭐⭐ |
| C: 第三方库（RichEditor） | 开箱即用 | 依赖风险、定制受限 | ⭐⭐⭐ |

**推荐方案 A**: 使用 AnnotatedString + 自定义 TextField，完全控制且无外部依赖。

### 决策 2: 拖拽排序实现方式

| 方案 | 优点 | 缺点 | 推荐 |
|------|------|------|------|
| **A: Compose Foundation DragAndDrop** | 原生支持、流畅 | API 较新、文档少 | ⭐⭐⭐⭐⭐ |
| B: 第三方库（reorderable） | 成熟稳定 | 依赖外部库 | ⭐⭐⭐⭐ |
| C: 手动实现触摸手势 | 完全可控 | 工作量大、易出 bug | ⭐⭐⭐ |

**推荐方案 A**: 使用官方 DragAndDrop API，配合 LongPressDraggableModifier。

### 决策 3: 国际化策略

| 方案 | 优点 | 缺点 | 推荐 |
|------|------|------|------|
| **A: Android 原生 strings.xml** | 系统级支持、自动切换 | 需维护多份文件 | ⭐⭐⭐⭐⭐ |
| B: 运行时动态加载 | 灵活 | 性能开销、复杂度高 | ⭐⭐⭐ |
| C: 仅英文硬编码 | 最简单 | 不满足需求 | ⭐ |

**推荐方案 A**: 标准的多语言资源管理，支持系统语言自动切换。

## ⚠️ 注意事项与风险

### 高风险项
1. **富文本编辑器** - AnnotatedString 光标管理复杂，需充分测试
2. **拖拽排序** - Compose DragAndDrop API 可能存在兼容性问题
3. **数据库迁移** - 新增 position 字段需处理旧数据

### 缓解措施
- 富文本编辑器采用渐进式实现，先支持基本格式再扩展
- 拖拽排序提供 fallback（长按弹出"上移/下移"菜单）
- 数据库 Migration 策略：默认按 ID 排序作为初始 position

### 性能考虑
- 动画使用 `animate*AsState()` 而非手动 Animation
- 拖拽时仅更新受影响项的位置
