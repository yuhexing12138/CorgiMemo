### 第 1 章：Android 开发环境搭建

#### 🎯 本章目标
完成 Android 开发环境的搭建，能够创建并运行第一个 Android 项目。

#### 1.1 需要安装的软件

| 软件 | 版本要求 | 下载地址 | 用途 |
|-----|---------|---------|------|
| Android Studio | 最新稳定版 | https://developer.android.com/studio | Android 官方 IDE |
| JDK | 17 或 21 | https://adoptium.net | Java 开发工具包 |
| Trae IDE | 最新版 | https://www.trae.ai | AI 辅助编程 |

#### 1.2 安装 JDK 17

**步骤 1：下载 JDK**
1. 访问 https://adoptium.net
2. 选择 **Eclipse Temurin 17 LTS**
3. 下载 Windows 版本的 MSI 安装包

**步骤 2：安装**
1. 双击下载的 MSI 文件
2. 一路点击"下一步"，使用默认安装路径（`C:\Program Files\Eclipse Adoptium\jdk-17`）
3. 勾选"将 JAVA_HOME 添加到系统环境变量"
4. 点击"安装"

**步骤 3：验证安装**
打开命令提示符（CMD），输入：
```bash
java -version
```
应该显示：
```
openjdk version "17.0.x" 202x-xx-xx
OpenJDK Runtime Environment Temurin-17...
```

#### 1.3 安装 Android Studio

**步骤 1：下载**
1. 访问 https://developer.android.com/studio
2. 点击"Download Android Studio"
3. 下载 Windows 版本的 exe 安装包

**步骤 2：安装**
1. 双击 exe 文件
2. 选择"Next"直到安装完成
3. **重要**：第一次启动时选择"Standard"安装类型
4. 选择暗色主题（推荐）
5. 等待组件下载完成（可能需要 10-30 分钟，取决于网络）

**步骤 3：创建模拟器**
1. 打开 Android Studio
2. 点击右上角的"Device Manager"
3. 点击"Create Device"
4. 选择 **Pixel 7**
5. 选择系统镜像：**Android 14.0 (API 34)**
6. 点击"Download"下载镜像（约 1GB）
7. 完成后点击"Finish"

#### 1.4 配置 Trae IDE

**步骤 1：安装 Android 插件**
1. 打开 Trae IDE
2. 点击左侧扩展图标（或按 `Ctrl+Shift+X`）
3. 搜索并安装以下插件：
   - **Kotlin**（Kotlin 语言支持）
   - **Android**（Android 开发支持）

**步骤 2：配置 JDK 路径**
1. 打开 Trae 设置（`Ctrl+,`）
2. 搜索 "java.home"
3. 设置为你的 JDK 安装路径：
   ```
   C:\Program Files\Eclipse Adoptium\jdk-17
   ```

#### 🆘 常见问题

**Q1: 模拟器启动很慢或卡顿？**
> A: 确保电脑开启了虚拟化技术（Intel VT-x 或 AMD-V）。在 BIOS 中开启。

**Q2: 提示 "SDK not found"？**
> A: 在 Android Studio 中打开 SDK Manager，安装 Android SDK Platform 34。

**Q3: 下载组件很慢？**
> A: 在 Android Studio 设置中配置国内镜像：
> - HTTP Proxy → 选择 "Auto-detect proxy settings"
> - 或者手动设置为：mirrors.tuna.tsinghua.edu.cn

#### 🎯 本章目标

#### 3.1 概念解释

**什么是 Room？**
> Room 是 Android 官方提供的 SQLite 数据库封装库。它让你可以用面向对象的方式操作数据库，不用写复杂的 SQL 语句。

**核心组件**：
- **Entity（实体）**：数据库表的结构定义（类似 Java 类）
- **DAO（数据访问对象）**：定义对数据库的操作方法（增删改查）
- **Database（数据库）**：数据库的入口，管理所有表

**TodoItem 实体类**：

### 数据表字段说明（替代原代码）

**待办事项表 (todo_items)**：
| 字段名 | 类型 | 作用 | 示例 |
|-------|------|------|------|
| id | 数字 | 唯一标识，自动增长 | 1, 2, 3... |
| title | 文字 | 待办标题，必填 | "开会讨论项目" |

| categoryId | 数字 | 所属分类 | 1=工作, 2=学习 |
| priority | 数字 | 优先级 | 0=低, 1=中, 2=高 |
| status | 数字 | 完成状态 | 0=待办, 1=已完成 |
| dueDate | 时间 | 截止日期 | 1704067200000 |
| reminderTime | 时间 | 提醒时间 | 1704063600000 |
| repeatType | 数字 | 重复类型 | 0=不重复 |
| createdAt | 时间 | 创建时间 | 系统自动生成 |
| updatedAt | 时间 | 更新时间 | 系统自动生成 |
| completedAt | 时间 | 完成时间 | 可为空 |

**💡 新手注意**：
- 时间字段存储的是毫秒时间戳，不是普通日期格式
- 数字类型的字段在数据库中占用空间更小，查询更快

**关键注解说明**：
- `@Entity`：标记这是一个数据库表
- `@PrimaryKey`：主键，`autoGenerate = true` 表示自增
- `@ColumnInfo`：指定列名（如果不指定，默认使用属性名）
- `@Index`：创建索引，加快查询速度

### 帧动画工作原理（替代原代码）

**什么是帧动画？**
> 帧动画就像翻书动画：每一页是一帧，快速翻动就形成了动画。
> - 1 帧 = 1 张图片
> - 帧率 = 每秒显示多少张图片（FPS）
> - 帧延迟 = 显示每帧的时长（毫秒）

**帧动画的关键参数**：
| 参数 | 作用 | 推荐值 |
|-----|------|-------|
| 帧数 | 动画包含的图片数量 | 3-8 张 |
| 帧延迟 | 每帧停留时间（毫秒） | 200-300ms |
| 是否循环 | 播放完是否从头开始 | 默认循环 |

**💡 新手注意**：
- 帧数越多动画越流畅，但图片越多占用内存越大
- 帧延迟太小（<100ms）会导致动画闪烁
- 建议先准备 3-5 张简单图片测试，熟悉后再增加帧数

**如果失败怎么办？**
- 动画不播放 → 检查图片资源是否正确放入 `res/drawable/`，资源名是否与代码中一致
- 动画闪烁 → 检查帧切换逻辑，确保使用了 `remember` 缓存状态，在 Trae SOLO 模式中输入："帧动画切换时有闪烁"
- 姿态不切换 → 检查 ViewModel 中的状态更新逻辑，确认 `StateFlow` 正确发射新值
- 粒子特效不显示 → 确认 `HeartParticleEffect` 组件的 `showHearts` 状态正确更新
- 内存持续增长 → 检查 `LaunchedEffect` 是否正确取消，避免重复启动协程
- 经验值不增加 → 检查完成任务时是否正确调用了经验值增加逻辑
- 等级不提升 → 检查升级阈值判断逻辑，确认经验值达到阈值后触发升级
- 装扮不显示 → 检查装扮资源是否正确放入 `res/drawable/`，路径是否正确
- 成就不解锁 → 检查成就检测触发时机，确认在每次完成任务后都调用了检测逻辑

#### 🆘 常见问题

**Q1: 动画播放卡顿？**
> A: 检查图片大小，建议每张图片不超过 200KB。可以使用在线工具压缩图片。

**Q2: 内存占用过高？**
> A: 使用 `remember` 缓存动画帧，避免重复加载。只在需要时加载图片资源。

**Q3: 动画切换不流畅？**
> A: 使用 `Crossfade` 或 `AnimatedContent` 实现过渡动画。

### 常见问题解决

#### 编译错误

**Q: 提示 "Unresolved reference: R"？**
> A: 清理并重建项目：Build → Clean Project → Rebuild Project

**Q: 提示 "Room schema export" 错误？**
> A: 在 build.gradle 中添加：
>
> ### 数据库迁移说明
>
> **什么是数据库迁移？**
> > 当你需要修改表结构（如增加字段）时，需要迁移。
> > 不迁移直接升级会导致数据丢失或崩溃。
>
> **💡 新手注意**：
> - 开发阶段可以用 `fallbackToDestructiveMigration()` 简单处理（会丢失数据）
> - 正式发布前必须写迁移脚本
> - 每次修改表结构前先问自己：能否用新增表代替修改？

#### 运行时错误

**Q: APP 启动崩溃？**
> A: 检查 Logcat 错误日志，常见原因：
> - 忘记在 AndroidManifest.xml 中声明 Application 类
> - Room 数据库版本变更但没有迁移
> - 依赖注入配置错误

**Q: 数据库升级后数据丢失？**
> A: 在 Database 类中增加版本号，并添加迁移：
>
> ### 数据库版本管理
>
> **版本号递进规则**：
> - 1 → 2：第一次修改表结构
> - 2 → 3：第二次修改表结构
> - 每个版本需要一个 Migration 类
>
> **💡 新手注意**：
> - 版本号只能增加，不能回退
> - 开发阶段可用 RoomDatabase.Builder 的 `.fallbackToDestructiveMigration()` 跳过迁移
> - 正式发布前一定要写 Migration，否则用户更新后会丢数据

### 术语速查表

> 💡 **使用说明**：这是给 Android 开发新手的术语解释。每个术语都有通俗类比和具体例子，帮助你理解 AI 提示词中的专业术语含义。

#### 一、开发工具与环境类

| 术语 | 通俗解释 | 具体例子 |
|-----|---------|---------|
| **JDK (Java Development Kit)** | Java 开发工具包，相当于"厨房用具套装"，里面包含编写、运行 Java 代码所需的所有工具 | 就像做菜需要锅碗瓢盆，写 Android 代码需要 JDK 的 `javac`（编译器）和 `java`（运行器） |

| **API (Application Programming Interface)** | 应用程序接口，相当于"点餐柜台"，告诉别人你能提供什么服务 | 比如你开发了一个天气查询功能，别人只需要调用 `getWeather(city)` 就能获取天气，不需要知道内部怎么实现 |
| **Gradle** | 构建工具，相当于"自动流水线"，帮你把代码编译成 APK | 就像工厂里的流水线，Gradle 自动处理：下载依赖 → 编译代码 → 打包APK → 生成签名 |

| **模拟器 (Emulator)** | 虚拟手机程序，相当于"手机模型"，在电脑上模拟运行 Android 系统 | 就像飞行员用飞行模拟器练习，开发者用模拟器测试 APP，不需要真手机也能开发 |
| **APK / AAB** | APP 安装包格式，APK=直营店，AAB=应用商城专用格式 | APK 就像可以直接安装的软件包（.exe）；AAB 是 Google Play 要求的格式，必须用它才能上架 Google 应用商店 |

#### 二、架构与代码组织类

| 术语 | 通俗解释 | 具体例子 |
|-----|---------|---------|
| **MVVM (Model-View-ViewModel)** | 一种代码组织方式，相当于"餐厅的三层结构"：厨房(Model)→传菜员(ViewModel)→餐桌(View) | Model 是做菜的厨房（数据），ViewModel 是传菜员（处理逻辑），View 是餐桌（显示界面），三者分工合作 |
| **ViewModel** | 页面数据管理者，相当于"餐厅服务员"，负责把数据整理好端给客人 | 当你在待办列表页面，`TodoListViewModel` 负责：从数据库取数据 → 筛选排序 → 准备好显示格式，供页面使用 |
| **Composable** | Compose 中的 UI 组件，相当于"乐高积木"，每个 Composable 是一块可组合的积木 | `Button()` 是一块积木，`Text()` 是一块积木，把它们拼在一起就是一个登录页面，就像玩乐高一样 |
| **StateFlow** | 状态流管理器，相当于"实时广播"，数据变化时自动通知订阅者 | 就像天气预报广播，你（订阅者）不用一直去问天气，广播会主动推送变化。`todoList.collectAsState()` 会在列表变化时自动更新界面 |
| **Repository** | 数据仓库管理员，相当于"超市采购员"，统一管理数据从哪来 | 你不需要关心数据是来自本地数据库还是网络云端，`TodoRepository` 帮你处理一切，就像去超市不用管货架怎么补货 |
| **Entity** | 数据库表结构定义，相当于"产品标签"，规定数据库里每条记录有哪些字段 | `TodoItem` Entity 定义了一条待办包含：标题、是否完成、创建时间，就像产品标签上写着：品名、生产日期、价格 |
| **DAO (Data Access Object)** | 数据访问对象，相当于"餐厅点菜系统"，定义对数据库的具体操作 | `@Query("SELECT * FROM todos")` 就像点菜系统里的"查询所有菜单"，你不需要自己翻找，告诉 DAO 要什么操作就行 |
| **Room** | Android 数据库库，相当于"智能文件柜"，帮你把数据库操作变得简单 | 传统写 SQLite 要几千行代码，用 Room 只需要定义 Entity + DAO，就像从手写账本升级到智能文件柜 |

| **DataStore** | 轻量级数据存储，相当于"便签本"，存设置项、用户偏好 | `PreferencesDataStore` 适合存开关状态、主题偏好，就像把"深色模式开关"状态写在便签上，随用随取 |

#### 四、SOLO 模式相关类

| 术语 | 通俗解释 | 具体例子 |
|-----|---------|---------|
| **SOLO 模式** | Trae IDE 的 AI 辅助模式，相当于"智能副驾驶"，帮你写代码、解释代码、修复错误 | Chat 模式是"问一句答一句"，SOLO 模式是"你说目的地，它帮你规划路线并开车" |
| **Plan 模式** | 规划模式，相当于"先画蓝图"，AI 先分析需求，列出详细的实现步骤 | 就像盖房子前先出设计图，Plan 模式让你先看到"这里要打地基→这里要砌墙→这里要装窗户"，确认后再施工 |

#### 五、数据库与网络类

| 术语 | 通俗解释 | 具体例子 |
|-----|---------|---------|
| **SQLite** | 轻量级数据库，相当于"抽屉式收纳盒"，每个抽屉是一个表 | 就像药盒有"西药抽屉""中药抽屉"，SQLite 数据库有"待办表""柯基表"，每个抽屉里放同类物品 |
| **WebDAV** | 网络文件同步协议，相当于"云端文件夹"，把本地文件和云端同步 | 就像 iCloud 同步照片，WebDAV 允许你把备忘录数据同步到 ownCloud、坚果云等私有云盘 |
| **JSON** | 数据格式标准，相当于"快递单"，用于前后端或客户端与服务端交换数据 | `{"title": "买菜", "done": false}` 就像快递单上的收件人、商品信息，双方约定好看得懂 |
| **Flow** | Kotlin 的响应式数据流，相当于"水龙头"，打开就有水流，关闭就停 | `flow { emit(1); emit(2) }` 就像打开水龙头，水（数据）就流出来，收集器在另一端接水 |
| **迁移 (Migration)** | 数据库升级脚本，相当于"搬家方案"，把旧数据安全转移到新结构 | APP 更新时数据库结构变了（加了字段），Migration 就是"先打包旧数据→拆掉旧柜子→装新柜子→放回数据" |

| 术语 | 通俗解释 | 具体例子 |
|-----|---------|---------|
| **Git** | 版本控制系统，相当于"时光机+游戏存档"，记录每次修改，可以回到任意版本 | 就像游戏存档，"今天下午3点的存档"坏了，可以回到"今天中午12点的存档"重新开始 |
| **commit** | 提交/存档点，相当于"游戏检查点"，把当前修改保存为一个快照 | `git commit -m "添加登录功能"` 就像在游戏里按 Save，下次可以从这里继续，不会丢掉进度 |
| **branch** | 分支，相当于"平行世界"，在不影响主线的情况下开发新功能 | `git checkout -b feature/corgi-animation` 就像进入"柯基动画"的平行世界，在里面折腾不会影响"主世界" |
| **merge** | 合并分支，相当于"平行世界融合"，把实验世界的成果合并回主线 | 当"柯基动画"平行世界完成测试后，用 `git merge` 把成果合并回主线，两个世界合二为一 |
| **Release 版本** | 正式发布版本，相当于"正式出版的书籍"，经过校对，不能随意修改 | Debug 是"草稿"，可以随便改；Release 是"定稿"，要经过测试、签名，才允许发布给用户 |

#### 七、动画与交互类

| 术语 | 通俗解释 | 具体例子 |
|-----|---------|---------|
| **帧动画 (Frame Animation)** | 逐帧播放图片，相当于"翻书动画"，快速翻页产生动感 | 柯基走路动画 = 连续播放 8 张图片，每张图片是走路的"分解动作"，快速翻页就变成连贯走路 |
| **触摸手势 (Gesture)** | 手指在屏幕上的动作，相当于"触屏指令"，有点击、滑动、长按等 | 双击柯基 = 两次快速触摸 = `doubleClick()` 手势识别 → 触发"打招呼"动画 |
| **状态机 (State Machine)** | 状态管理系统，相当于"交通信号灯"，规定什么状态可以转到什么状态 | 柯基状态：吃饭→走路→睡觉，只有在"睡觉"状态才能触发"叫醒"动作，不能从"吃饭"直接跳到"睡觉" |
| **缓动函数 (Easing)** | 动画节奏曲线，相当于"运动风格"，决定动画是匀速、加速还是弹跳 | 弹跳效果 = `FastOutSlowInEasing`，像球落地又弹起；匀速 = `LinearEasing`，像匀速跑步 |

### 项目开发规范

#### 二、设计规范

UI设计采用Material3组件库，必须支持深色模式，主色调为暖橙色FF9A5C，卡片和按钮圆角为16dp，字体层级遵循Material排版规范。

架构采用MVVM四层结构：UI层使用Jetpack Compose负责界面展示，ViewModel层使用StateFlow管理状态，Repository层处理数据协调，Data层使用Room和OkHttp处理存储和网络请求。

命名规范要求：Composable函数使用大驼峰如HomeScreen，ViewModel类以功能加ViewModel命名如HomeViewModel，Repository类以实体加Repository命名如TodoRepository，Entity类直接使用实体名如TodoItem，DAO接口以实体加Dao命名如TodoDao。

#### 三、代码规范

所有类和公共方法必须有中文注释，复杂逻辑需要行内注释说明。使用StateFlow管理UI状态，使用remember缓存Composable状态，避免在Composable中直接修改状态。耗时操作必须使用协程，在ViewModel中使用viewModelScope，在Composable中使用LaunchedEffect。所有网络请求和数据库操作必须有try-catch异常处理，提供用户友好的错误提示并记录错误日志。

禁止行为包括：禁止使用硬编码字符串应使用stringResource，禁止在主线程执行耗时操作应使用协程，禁止在ViewModel中直接操作UI应通过StateFlow通知，禁止忽略空安全应使用问号运算符处理。

#### 四、Git提交规范

提交信息格式为type冒号subject，可选body。Type类型包括feat表示新功能如实现待办列表，fix表示修复Bug如修复列表不刷新，refactor表示重构代码，docs表示文档更新，style表示代码格式，test表示测试相关。

分支命名规范：功能分支使用feature斜杠功能名如feature斜杠corgi-animation，修复分支使用fix斜杠问题描述如fix斜杠crash-on-save，发布分支使用release斜杠版本号如release斜杠v1.0.0。

#### 五、功能优先级规范

遵循MoSCoW方法：M表示必须实现的核心功能如待办CRUD和柯基陪伴，S表示应该实现的重要功能如数据同步和节日问候，C表示可以实现的锦上添花功能如桌面小部件和成就系统，W表示暂不实现的后续迭代功能如社交功能和多语言。

#### 六、质量检查清单

代码质量要求有详细中文注释，无编译错误和警告，遵循Kotlin编码规范，无硬编码字符串。功能完整性要求实现所有需求功能点，正常和异常流程都有处理，UI适配深色模式，支持不同屏幕尺寸。测试验证要求在模拟器中运行成功，无崩溃和ANR，内存无泄漏，已提交Git。
