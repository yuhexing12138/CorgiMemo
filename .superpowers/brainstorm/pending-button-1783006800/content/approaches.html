📋 待完成按钮 - 方案对比

方案 1:复用 PinnedDivider 模式(推荐)

直接镜像 PinnedSectionHeader + PinnedDivider 的实现方式

核心思路

  • 新增 PendingSectionHeader 组件,调用 CollapsibleSectionHeader
  • 新增 DisplayItem.PendingDivider 变体
  • 在 displayItems 中动态插入
  • 位置逻辑:置顶 ≤ 3 时插在最前;> 3 时插在置顶区后
@Composable fun PendingSectionHeader(...) = CollapsibleSectionHeader( label = "待完成", color = Color.Blue, // 蓝色 ... )

优点

  • 改动最小、风险最低、与现有代码一致
  • 复用 CollapsibleSectionHeader,无重复代码
  • 复用现有 4 处修改模式(DisplayItem/状态订阅/构建/渲染)
  • 复用现有 CorgiPreferences 持久化模式
  • 复用现有单元测试模式

缺点

  • displayItems 构建逻辑变复杂(三段式 if/else)
  • 未解决根本性"section header 模式重复"问题
  • 未来新增其他 section(如"今日到期")需再次重复

方案 2:SectionHeader 抽象层

抽出一个统一的 SectionHeader 描述,运行时动态构建

核心思路

  • 定义 data class SectionSpec(label, count, color, showKey, isExpanded)
  • displayItems 接收 List<SectionSpec> 动态生成
  • ViewModel 暴露 sections: StateFlow<List<SectionSpec>>
  • UI 层只负责按 spec 渲染,不知道具体是置顶/待完成/已完成
data class SectionSpec( val id: String, // "pinned" | "pending" | "completed" val label: String, val count: Int, val color: Color, val isExpanded: Boolean, val items: List<TodoItem> = emptyList() )

优点

  • 声明式,UI 层逻辑最简
  • 易于扩展新的 section 类型
  • Section 顺序/位置动态调整更容易
  • ViewModel 主导布局决策,UI 纯渲染

缺点

  • 需要重构已完成的 PinnedDivider 实现
  • SectionSpec 设计需要考虑拖拽、key 等细节
  • 回归风险较高(影响现有置顶按钮)
  • 本次需求范围内属"过度设计"

🎯 我的推荐:方案 1

理由:
1. 本次需求仅新增 1 个 section 按钮,与方案 1 模式 1:1 对应
2. 改动最小、风险最低、便于 review 和测试
3. 保持与刚完成的"置顶按钮"完全对称,代码风格统一
4. 未来真要扩展(如"今日到期"等),可以一次性重构为方案 2(YAGNI)
