⚙️ 底部导航栏 - 技术实现方案

3 种架构方案的权衡分析与推荐

A

Navigation Compose + 自定义组件 (推荐)

核心思路：使用 Navigation Compose 管理多页面路由，完全自定义 BottomNavigationBar 组件

✅ 优势：
• 完全控制 UI 和动画（中央按钮突出、气泡展开）
• 符合项目现有架构（已用 Navigation Compose）
• 易于扩展新 Tab 页面
• 动画性能优化空间大

❌ 劣势：
• 需要自行管理组件状态
• 代码量相对较多（约 400-500 行）
适用场景：需要高度定制化 UI 和复杂动画
B

Material3 NavigationBar + FAB 扩展

核心思路：使用 Material3 内置 NavigationBar，通过自定义 Slot 插入中央 FAB 按钮

✅ 优势：
• 符合 Material Design 3 规范
• 代码量较少（约 250-300 行）
• 内置无障碍支持
• 系统级主题适配

❌ 劣势：
• 中央按钮气泡动画实现困难
• 突出 8dp 效果需 hack
• 定制灵活性受限
适用场景：标准 UI 需求，无需复杂动画
C

纯自定义 Canvas 绘制

核心思路：使用 Compose Canvas API 完全手绘底部导航栏

✅ 优势：
• 极致灵活性和性能
• 可实现任意形状和动画
• 渲染性能最优

❌ 劣势：
• 开发成本极高（800+ 行）
• 需要手动处理点击检测
• 无障碍支持需完全自建
• 维护成本高
适用场景：特殊形状或极致性能需求

💡 我的推荐：方案 A

理由：CorgiMemo 项目已经使用 Navigation Compose，方案 A 可以无缝集成现有架构。虽然代码量比 B 多一些，但能够完美实现设计文档中的所有要求（中央按钮突出 8dp、扇形气泡动画、200ms 弹出时序）。方案 B 虽然代码少，但无法优雅地实现气泡展开效果；方案 C 则过度工程化。
