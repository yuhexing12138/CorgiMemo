# 下拉刷新快速下滑卡住修复设计

> 创建日期：2026-07-08
> 类型：Bug 修复
> 影响范围：待办首页（HomeScreen）的下拉刷新体验

## 1. 问题描述

### 1.1 现象
用户在待办首页用手指快速下滑时，下拉刷新指示器**有时会卡在「下拉刷新」或「释放刷新」两个状态不恢复**：
- 即使手指已经离开屏幕，指示器仍持续显示
- 直到用户再次触摸并滚动列表，指示器才回弹消失
- 用户在「已完成区折叠」等列表较短场景下尤其容易触发

### 1.2 影响
- 视觉卡顿，体验割裂
- 占用列表顶部空间，阻挡第一个待办项
- 反复出现会让用户怀疑"下拉刷新坏了"

## 2. 原因分析

经过对 [PullRefreshStateHolder.kt](file:///c:/Users/EDY/Desktop/CorgiMemo/app/src/main/java/com/corgimemo/app/ui/components/PullRefreshStateHolder.kt) 的代码分析，确认是 **3 个互相叠加**的原因：

### 2.1 根因：Compose NestedScroll 的 `onPreFling` 触发条件

`NestedScrollConnection.onPreFling` **只有在子组件（LazyColumn）自身 fling 时才会被父组件回调**。

- 当 `LazyColumn` 在列表底部时，自身无法向上滚动
- 用户从底部快速下滑时，LazyColumn 不会 fling
- 父组件的 `onPreFling` 永远不会被触发
- 而回弹/刷新逻辑全部写在 `onPreFling` 内部
- → **`state` 永远停在 `PULLING`，`pullOffset` 不会归零**

这是最核心的根因，在以下场景必现：
- 列表项数 < 一屏（用户从底部快速下滑）
- 用户已滚动到底部再快速下滑
- "收齐已完成" 等折叠状态

### 2.2 次因：`onPreFling` 是 suspend 协程但没用 try-finally

旧实现：
```kotlin
override suspend fun onPreFling(available: Velocity): Velocity {
    if (state != PullRefreshState.PULLING) return Velocity.Zero
    if (pullOffset >= refreshThresholdPx) {
        state = PullRefreshState.REFRESHING
        onRefresh()
        animate(pullOffset, refreshThresholdPx, ...) { pullOffset = value }  // 阻塞 300ms
        rawPullOffset = ...
    } else {
        state = PullRefreshState.RELEASING
        animate(pullOffset, 0f, ...) { pullOffset = value }  // 阻塞 300ms
        pullOffset = 0f
        rawPullOffset = 0f
        state = PullRefreshState.IDLE
    }
    return Velocity.Zero
}
```

- `animate(...)` 会**阻塞 fling 协程 300ms**
- **没用 `try-finally` 保护**
- 如果用户在 300ms 内产生新的手势（连续快速下滑），`NestedScrollDispatcher` 可能**取消**当前 fling 协程
- `animate` 被取消 → 后续 `state = IDLE` 不会执行
- 状态卡在 `RELEASING` 或 `REFRESHING`

### 2.3 第三因：阻尼曲线在动画中断时会断裂

`rawPullOffset` 是累积式的（`rawPullOffset += available.y`），设计时假设 `pullOffset = computeDampedOffset(rawPullOffset)`。

- 当 releaseJob 动画进行到一半被取消，`pullOffset` 停在中间值（如 15）
- 此时 `rawPullOffset` 仍为 0（重置发生在动画末尾）
- 用户再次下拉：`rawPullOffset += available.y` 从 0 开始
- 但 `pullOffset` 是从 15 开始
- → 阻尼曲线对应关系**断裂**，再次下拉的位置出现跳变

## 3. 修复方案

采用 **方案 A + D 组合**：

| 方案 | 思路 | 作用 |
|------|------|------|
| **A. 松手立即回弹** | 在外层 Box 添加 `Modifier.pointerInput` 监听 up 事件，松手时直接调用 `onRelease()` 启动回弹，不再依赖 `onPreFling` | 解决根因（2.1） |
| **D. try-finally 保护** | `onPreFling` 内部也用 `onRelease()` 委托，改为 `scope.launch` 异步执行，try-catch `CancellationException` 透传 | 解决次因（2.2 + 2.3） |

## 4. 实施细节

### 4.1 PullRefreshStateHolder.kt 重构

#### 4.1.1 统一释放入口

新增公共方法 `onRelease()` 作为外部 pointerInput 监听到 up 事件时的入口：

```kotlin
fun onRelease() {
    if (state != PullRefreshState.PULLING) return  // 非 PULLING 状态是 no-op
    doRelease()
}
```

#### 4.1.2 提取统一的 doRelease 私有方法

将原本在 `onPreFling` 内的回弹/刷新逻辑提取为 `doRelease()`，并改用 `scope.launch` 异步执行：

```kotlin
private fun doRelease() {
    releaseJob?.cancel()  // 取消任何旧的 release 任务

    val startOffset = pullOffset
    val goToRefresh = startOffset >= refreshThresholdPx

    releaseJob = scope.launch {
        try {
            if (goToRefresh) {
                state = PullRefreshState.REFRESHING
                onRefresh()
                animate(startOffset, refreshThresholdPx, tween(300)) { pullOffset = it }
                rawPullOffset = computeRawOffsetForDamped(refreshThresholdPx, maxPullHeightPx)
            } else {
                state = PullRefreshState.RELEASING
                animate(startOffset, 0f, tween(300)) { pullOffset = it }
                pullOffset = 0f
                rawPullOffset = 0f
                state = PullRefreshState.IDLE
            }
        } catch (e: CancellationException) {
            // 协程被取消时，状态由调用方决定
            // 此处不强行重置 state，避免与新事件竞争
            throw e
        }
    }
}
```

#### 4.1.3 onPreFling 委托

```kotlin
override suspend fun onPreFling(available: Velocity): Velocity {
    if (state != PullRefreshState.PULLING) return Velocity.Zero
    onRelease()  // 委托给统一入口，异步执行不阻塞 fling 协程
    return Velocity.Zero
}
```

#### 4.1.4 onPostScroll 阻尼曲线校准

解决原因 2.3：每次 onPostScroll 开始时，**先根据当前 pullOffset 反推 rawPullOffset**，再累加：

```kotlin
override fun onPostScroll(...): Offset {
    if (state == PullRefreshState.REFRESHING) return Offset.Zero
    if (available.y > 0f) {
        cancelRelease()
        // 关键：先根据当前 pullOffset 重新校准 rawPullOffset
        rawPullOffset = computeRawOffsetForDamped(pullOffset, maxPullHeightPx)
        rawPullOffset += available.y
        val newOffset = computeDampedOffset(rawPullOffset, maxPullHeightPx)
        ...
    }
}
```

#### 4.1.5 onPreScroll 同步 rawPullOffset

`onPreScroll` 原本就有这段逻辑，确认保留：

```kotlin
val newOffset = (pullOffset + delta).coerceAtLeast(0f)
pullOffset = newOffset
rawPullOffset = computeRawOffsetForDamped(newOffset, maxPullHeightPx)
```

### 4.2 HomeScreen.kt 添加 pointerInput 兜底

在 [HomeScreen.kt](file:///c:/Users/EDY/Desktop/CorgiMemo/app/src/main/java/com/corgimemo/app/ui/screens/home/HomeScreen.kt) 外层 Box 上添加 `pointerInput` 监听 up 事件：

```kotlin
Box(
    modifier = Modifier
        .fillMaxSize()
        .nestedScroll(pullRefreshState.nestedScrollConnection)
        // 兜底监听"松手"事件：解决列表底部快速下滑时
        // onPreFling 不会被触发的卡住问题
        .pointerInput(pullRefreshState) {
            awaitEachGesture {
                // 监听但不消费 down 事件（不阻塞 LazyColumn 滚动）
                awaitFirstDown(requireUnconsumed = false)
                // 循环等待所有指针抬起（支持多指）
                do {
                    val event = awaitPointerEvent()
                } while (event.changes.any { it.pressed })
                // 手指完全抬起 → 触发松手处理
                pullRefreshState.onRelease()
            }
        }
) { ... }
```

**关键设计**：
- `awaitFirstDown(requireUnconsumed = false)`：不要求 down 事件未消费，避免与 LazyColumn 的 draggable 冲突
- `event.changes.any { it.pressed }`：等待所有指针抬起，天然支持多指
- `onRelease()` 内部有 `if (state != PULLING) return` 保护，普通点击/拖动 todo 不会触发回弹

## 5. 修复前后行为对比

| 场景 | 修复前 | 修复后 |
|------|--------|--------|
| 列表底部快速下滑（无 fling） | ❌ onPreFling 不触发，卡在 PULLING | ✅ pointerInput 兜底，up 时调用 onRelease |
| 快速连续下拉 + 松手 | ❌ 协程被取消后 state 卡在中间 | ✅ releaseJob 统一管理 + try-catch 透传 |
| 释放期间 rawPullOffset 错位 | ❌ 阻尼曲线断裂，再次下拉跳变 | ✅ onPostScroll 校准 rawPullOffset |
| onPreFling 阻塞 300ms | ❌ fling 协程被取消会丢失状态收尾 | ✅ 异步执行，不阻塞 fling 协程 |

1. **节流 onPostScroll**：当前每次 scroll 事件都调 `cancelRelease()`，可合并为 50ms 节流

## 8. 风险评估

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|----------|
| pointerInput 与 LazyColumn 滚动冲突 | 低 | 中 | `awaitFirstDown(requireUnconsumed = false)` 不消费事件；onRelease 内部有 PULLING 状态保护 |
| scope.launch 异步执行后 onPreFling 返回时机 | 低 | 低 | onRelease 是非 suspend，立即返回不阻塞 |
| 连续快速下拉 race condition | 中 | 低 | releaseJob 统一管理 + try-catch + 阻尼曲线校准三道防护 |
| 多指操作导致 up 事件误判 | 极低 | 低 | `event.changes.any { it.pressed }` 天然支持多指 |

## 9. 总结

✅ **核心修复**：用 `Modifier.pointerInput` 监听 up 事件作为 `onPreFling` 的兜底，从根上解决"列表底部快速下滑卡住"问题

✅ **健壮性提升**：用 `scope.launch` + `releaseJob` 统一管理动画，避免 fling 协程被取消时状态机卡死

✅ **体验一致性**：阻尼曲线在动画中断场景下不再断裂，位置跳变问题解决
