## 一、项目背景与目标

### 1.1 背景

当前 `SpecialDateEditScreen` 同时承担"新建"与"编辑"两个职责，包含完整的 10+ 字段表单（标题/日期/分类/计时模式/重复类型/提醒设置/备注/标签/图片/关联）。该设计存在以下问题：

1. **新建流程过重**：用户仅想快速记录一个日期时，需要面对大量不相关的字段，认知负担重。
2. **移动端表单最佳实践**：参考 iOS Reminders、滴答清单等主流应用，新建流程通常采用"先核心后细节"的两步式（先输入关键信息，再补充细节）。
3. **与产品演进方向不匹配**：用户希望分阶段实现，先落地"核心 4 行"页面，"日期卡片设置页"（含备注/标签/图片/关联/计时/重复/提醒）后续实现。

### 1.2 目标

- 新建一个 `SpecialDateQuickCreateScreen`（日期新建专用页面），仅保留 4 行核心功能：头像、名称、日期、类型、置顶、关联。
- 扩展 `DateCategory` 枚举从 4 个固定值到 7 个固定值 + "自定义"输入。
- 复用 `ReminderPickerBottomSheet` 作为日期选择器（保留完整功能，后续再删减）。
- 所有"功能开发中"占位统一使用 Snackbar。
- 编辑模式（点击日期卡片）本次**不变**，仍走旧版 `SpecialDateEditScreen`。

### 1.3 不在本设计内

- 编辑模式（点击日期卡片）的页面重构 —— 后续任务实现"日期卡片设置页面"。
- `SpecialDateEditScreen` 的删除或清理 —— 保留以供编辑模式临时使用。
- "下一步"按钮的真实跳转逻辑 —— 当前阶段仅显示占位 Snackbar。
- 自定义类型的 emoji 图标支持 —— 当前阶段仅在弹窗内显示 7 个固定 emoji。
- 头像选择、相册选择、emoji 选择等具体功能 —— 当前阶段仅显示占位 Snackbar。

## 二、设计方案选型

### 2.1 方案对比

| 方案 | 描述 | 优点 | 缺点 | 推荐度 |
|------|------|------|------|--------|
| **方案 A：新建 SpecialDateQuickCreateScreen（已选）** | 新建 `SpecialDateQuickCreateScreen.kt`，`SpecialDateEditScreen` 保留不动（编辑模式暂用） | 边界清晰、新旧页面隔离、新页面可独立测试和后续清理 | 需要修改 `SpecialDateScreen` 的 FAB 路由；新增 1 个 Screen 对象 | ⭐⭐⭐⭐⭐ |
| 方案 B：保留 `SpecialDateEditScreen` 类名（重载） | `SpecialDateEditScreen` 文件保留，`specialDateId == null` 时渲染新"4 行布局"，否则渲染旧"全功能布局" | 路由零改动，调用方零改动 | 单文件多布局，文件膨胀；条件分支污染，复杂度上升；不利于后续清理 | ⭐⭐ |
| 方案 C：完全拆分 + 重新设计路由 | 新建 `SpecialDateQuickCreateScreen` 和占位版 `SpecialDateCardSettingScreen`；编辑模式跳占位页 | 完整流程视角；为后续编辑页重构铺路 | 范围最大，本次需求"只重构新建"，编辑模式被"抢占"到占位页不必要 | ⭐⭐⭐ |

### 2.2 选型结论

采用**方案 A：新建 SpecialDateQuickCreateScreen**。理由：

1. 与"只重构新建"需求严格对齐，不影响编辑模式。
2. 新旧页面隔离，未来清理 `SpecialDateEditScreen` 时不会影响新建流程。
3. 路由调整范围最小（仅 FAB 跳转 + 新增 1 个 Screen 对象）。
4. 与项目内其他"新建专用"页面（如 `InspirationEditScreen`）的命名风格一致。

## 四、数据流与状态管理

### 4.1 路由

| 路径 | 用途 | 备注 |
|------|------|------|
| `date_create` | 新建（4 行核心功能页） | **本次新增** |
| `date_edit` | 编辑（旧版全功能页，新建模式暂用） | 保留 |
| `date_edit/{id}` | 编辑（带 ID 路由） | 保留 |

### 4.2 Local 状态（页面级 `remember`）

| 状态名 | 类型 | 初始值 | 用途 |
|--------|------|--------|------|
| `title` | `String` | `""` | 名称输入框 |
| `targetDate` | `Long?` | `null` | 目标日期时间戳 |
| `selectedCategory` | `DateCategory` | `DateCategory.OTHER` | 当前选中的固定类型 |
| `customCategory` | `String?` | `null` | 自定义类型输入（如果有） |
| `isPinned` | `Boolean` | `false` | 置顶开关状态 |
| `showDatePicker` | `Boolean` | `false` | 日期选择弹窗显示 |
| `showTypePicker` | `Boolean` | `false` | 类型选择弹窗显示 |
| `dateRowText` | `String` | `"无"` | 日期行右侧显示文本（选中后变为日期字符串） |
| `typeRowText` | `String` | `"请选择"` | 类型行右侧显示文本（选中后变为类型名） |
| `relationRowText` | `String` | `"+ 添加"` | 关联行右侧显示文本（始终为占位） |

**表单验证（本期范围）**：
- **仅 UI 焦点态样式**：输入框获焦时显示 Material3 默认高亮边框
- **不强制必填校验**：因为"下一步"按钮当前不真正保存数据（仅显示"功能开发中"占位 Snackbar），无需拦截用户操作
- **后续任务**：当"下一步"按钮实现真实跳转/保存时，再补充必填校验逻辑

> **不写入 ViewModel**：本次范围仅 UI 展示，不涉及实际数据持久化。后续"日期卡片设置页面"实现时再考虑数据提交。

### 4.3 数据流

```
FAB (+) 点击
    ↓
MainScreen / SpecialDateScreen
    ↓ navController.navigate("date_create")
SpecialDateQuickCreateScreen
    ├─ Local 状态: title / targetDate / selectedCategory / isPinned
    ├─ 点击"日期"  →  ReminderPickerBottomSheet (复用现有)
    │                 → onConfirm(date, hour, minute, repeat, cal, due)
    │                 → 仅使用 date 部分
    ├─ 点击"类型"  →  DateTypePickerBottomSheet (新建)
    │                 → onSelected(category: DateCategory, customName: String?)
    │                 → 当 customName != null 时存为 "CUSTOM:$customName"
    ├─ 点击"置顶"  →  Switch 切换
    ├─ 点击"关联"  →  Snackbar "功能开发中"
    ├─ 点击头像  →  Snackbar "功能开发中"
    └─ 点击"下一步"  →  Snackbar "功能开发中"（不真正保存）
```

## 七、数据存储

### 7.1 DateCategory 枚举扩展

**枚举值清单（7 个固定 + 保留 OTHER 兼容）**：

```kotlin
enum class DateCategory(val displayName: String, val emoji: String) {
    BIRTHDAY("生日", "🎂"),
    ANNIVERSARY("纪念日", "💕"),
    HOLIDAY("节日", "🎉"),
    LIFE("生活", "🌱"),
    STUDY("学习", "📚"),
    WORK("工作", "💼"),
    ENTERTAINMENT("娱乐", "🎮"),
    OTHER("其他", "📅")  // 保留以兼容数据库中已有数据
}
```

### 7.2 自定义类型存储格式

| 来源 | 存储格式 | 示例 |
|------|----------|------|
| 固定枚举 | 直接使用枚举 `name` | `category = "BIRTHDAY"` |
| 自定义输入 | `"CUSTOM:" + 输入内容` | `category = "CUSTOM:旅行"` |
| 数据库旧数据 | 原 `OTHER` 等 | 保持兼容，不修改 |

### 7.3 解析逻辑

```kotlin
// SpecialDateViewModel.groupByDisplayDates
val category = runCatching {
    val raw = date.category
    if (raw.startsWith("CUSTOM:")) {
        // 自定义类型：分组时归到 OTHER（不影响颜色逻辑）
        DateCategory.OTHER
    } else {
        DateCategory.valueOf(raw)
    }
}.getOrDefault(DateCategory.OTHER)
```

> **注意**：本次仅 UI 层扩展枚举，**不修改数据库 Schema**（`SpecialDate.category` 字段类型仍为 String）。新枚举值通过字符串直接写入现有字段。
