## 一、项目背景与目标

### 1.1 背景

当前 `SpecialDateScreen` 存在以下问题：

1. **分组信息冗余**：使用"即将到来/正在纪念/已过期"三组 + `DateGroupHeader` 占位，每个分组上方都有一行小标题，垂直空间浪费。
2. **时间信息静态**：现有 `SpecialDateCard` 仅展示"剩余 X 天"，没有"年/天/时/分/秒"动态时间格式。
3. **时间不实时**：卡片的"剩余/已过"天数通过 `remember(item.date)` 缓存，不随时间变化，用户长时间停留页面时数据会过期。
4. **左滑交互缺失**：当前日期卡片无左滑操作，与待办卡片体验不一致。
5. **无软删除能力**：现有 `delete()` 是物理删除 + 级联清理，无中间状态可恢复。

### 1.2 目标

- 重新设计日期卡片组件，对齐 UI 设计规范（圆角 20dp、间距 8dp、卡片 16dp 内边距）。
- 引入"年/天/时/分/秒"动态时间格式，逐级显示到秒。
- 实现页面级单一 ticker 协程，1Hz 实时刷新所有可见卡片时间信息。
- 复用 `SwipeableTodoBox` 实现三按钮左滑（置顶/归档/删除），与待办卡片保持一致的飞书风格级联动画。
- 引入 `isArchived` 软删除字段 + 归档 Snackbar 3 秒撤回机制（"已归档"入口 UI 暂不做，先留接口）。
- 保留搜索框、`DateCategory`、置顶/编辑/删除等已有交互。

### 1.3 不在本设计内

- "已归档"入口的 UI 位置（用户已说明后续再确定）。
- `SpecialDateEditScreen` 编辑表单的字段调整。
- `SpecialDateRelation` 关联功能的改造。
- `DateCategoryPicker` / `DateGroupHeader` / `CountModeSwitch` / `RepeatTypePicker` / `ReminderSetting` / `SelectableChip` 等旧组件的清理（保留可作废文件但**不删除**，留待后续统一清理）。

## 二、设计方案选型

### 2.1 方案对比

| 方案 | 描述 | 优点 | 缺点 | 推荐度 |
|------|------|------|------|--------|
| 方案 A：**最小侵入式重构（已选）** | 复用 `SwipeableTodoBox` + `CollapsibleSectionHeader`，仅新增 `onArchiveClick` 入参；原地重构 `SpecialDateCard.kt` | 风险最低、改动范围集中在 `date` 包内、复用现成的 `LocalContentIndication` 与 `ElasticOutEasing` | 与待办耦合 SwipeableTodoBox 的 API 形态，长期会有限制 | ⭐⭐⭐⭐⭐ |
| 方案 B：完全解耦 | 新建 `SwipeableDateBox`、`DateListCard`、`DateGroupViewModel` | 日期页独立演进空间大 | 与 `SwipeableTodoBox` 90% 代码重复、本次任务范围显著扩大 | ⭐⭐⭐ |
| 方案 C：抽象通用 `SwipeableCardBox` | 重构 `SwipeableTodoBox` 为参数化通用组件 | 彻底消除重复 | 同时改动待办页代码，回归风险上升 | ⭐⭐⭐ |

### 2.2 选型结论

采用**方案 A：最小侵入式重构**。理由：

1. 与"UI 重构"诉求一致：本次需求是日期页 UI 升级，不涉及跨页重构。
2. 现有 `SwipeableTodoBox` 已是参数化设计（`onPinClick` / `onDeleteClick`），新增 `onArchiveClick` 入参即可支持归档，**不需要新建组件文件**。
3. 复用 `LocalContentIndication`（水波纹开关）和 `ElasticOutEasing`（弹性动画），保持与待办卡片视觉一致。
4. 唯一新建文件是 `DateSectionHeader.kt`（~30 行包装），无新组件、无新框架。

```kotlin
private val MIGRATION_33_34 = object : Migration(33, 34) {
    override fun migrate(db: SupportSQLiteDatabase) {
        // 1. 新增 isArchived 字段，默认值 0（与 @ColumnInfo defaultValue 一致）
        db.execSQL(
            "ALTER TABLE special_dates ADD COLUMN isArchived INTEGER NOT NULL DEFAULT 0"
        )
        // 2. 加索引：isArchived 过滤是主页查询常态
        db.execSQL(
            "CREATE INDEX IF NOT EXISTS index_special_dates_isArchived " +
            "ON special_dates(isArchived)"
        )
    }
}
```

### 5.2 `groupedDates` 重新实现

```kotlin
val groupedDates: StateFlow<Map<DateGroup, List<DisplayDate>>> =
    combine(specialDates, _searchQuery) { dates, query ->
        // 1) 只取未归档
        val active = dates.filter { !it.isArchived }
        // 2) 搜索过滤
        val filtered = if (query.isBlank()) active
                       else active.filter { it.title.contains(query, ignoreCase = true) }
        // 3) 按新规则分组
        filtered.groupBy { date ->
            val todayStart = todayStartMillis()
            when {
                date.countMode == 0 && date.targetDate >= todayStart -> DateGroup.COUNTDOWN
                date.countMode == 1 && date.targetDate <= todayStart -> DateGroup.COUNTUP
                date.countMode == 0 && date.targetDate <  todayStart -> DateGroup.EXPIRED
                else -> DateGroup.COUNTUP  // countMode=1 但 targetDate 仍在未来：正计时
            }
        }.mapValues { entry ->
            entry.value.map { it.toDisplayDate() }  // 转换为精简的 DisplayDate
        }
    }.stateIn(viewModelScope, SharingStarted.WhileSubscribed(5000), emptyMap())
```

`todayStartMillis()` 工具函数返回当天 0 点的毫秒值（与 `DateScreen.kt` 中 `calculateDaysRemaining` 一致算法）。

### 6.2 `DateSectionHeader.kt`（新增，~30 行）

```kotlin
@Composable
fun DateSectionHeader(
    group: DateGroup,
    count: Int,
    isExpanded: Boolean,
    onToggle: () -> Unit,
    modifier: Modifier = Modifier
) {
    val (label, color) = when (group) {
        DateGroup.COUNTDOWN -> "倒计时" to Color(0xFFFF9A5C)   // 主色
        DateGroup.COUNTUP   -> "正计时" to Color(0xFF7EC8A0)   // 柔和绿
        DateGroup.EXPIRED   -> "已过期" to Color(0xFF999999)   // 提示色
    }
    CollapsibleSectionHeader(
        label = label,
        count = count,
        isExpanded = isExpanded,
        color = color,
        onClick = onToggle,
        expandedLabel = "收起$label",
        collapsedLabel = "展开$label",
        modifier = modifier
    )
}
```

**默认状态**：
- `COUNTDOWN` 默认展开（true）
- `COUNTUP` 默认展开（true）
- `EXPIRED` 默认折叠（false）

### 6.6 时间格式化与实时刷新

#### 6.6.1 格式化函数

```kotlin
/**
 * 把毫秒差值格式化为 "X年 X天 X时 X分 X秒"，从最大有效单位逐级显示到秒。
 * 例：
 * - 366_1000ms → "1年 1天 0时 0分 1秒"
 * - 5天3时   → "5天 3时 0分 0秒"
 * - 不足1时 → "4分 5秒"
 * - =0      → "0秒"（兜底）
 *
 * 注：使用"365天=1年"简化计算（不区分闰年），与现有 calculateDaysRemaining 一致。
 */
internal fun formatDuration(millis: Long): String {
    val totalSec = (millis / 1000).coerceAtLeast(0L)
    val sec = (totalSec % 60).toInt()
    val min = (totalSec / 60 % 60).toInt()
    val hr  = (totalSec / 3600 % 24).toInt()
    val day = (totalSec / 86400 % 365).toInt()
    val yr  = (totalSec / (86400 * 365)).toInt()
    return when {
        yr  > 0 -> "$yr 年 $day 天 $hr 时 $min 分 $sec 秒"
        day > 0 -> "$day 天 $hr 时 $min 分 $sec 秒"
        hr  > 0 -> "$hr 时 $min 分 $sec 秒"
        min > 0 -> "$min 分 $sec 秒"
        else    -> "$sec 秒"
    }
}
```

#### 6.6.2 实时刷新（单一 ticker）

在 `SpecialDateScreen.kt` 顶层：

```kotlin
var nowMs by remember { mutableStateOf(System.currentTimeMillis()) }
LaunchedEffect(Unit) {
    while (true) {
        nowMs = System.currentTimeMillis()
        delay(1000L)
    }
}
```

- **单 ticker 协程**：整页只有 1 个 `LaunchedEffect` 在跑。
- **防协程泄漏**：离开 `Composable` 自动取消。
- **`SpecialDateCard` 接收 `nowMs` 参数**，不再各自开 `LaunchedEffect`。
- 性能：每秒 1 次 state 读取，触发持有 `nowMs` 的 Composable 重组；卡片多时可接受。

#### 6.6.3 已过/剩余计算

`SpecialDateCard` 内：
```kotlin
val diff = date.targetDate - nowMs
val isFuture = diff >= 0
Text(
    text = (if (isFuture) "还有" else "已经") + " " + formatDuration(kotlin.math.abs(diff)),
    color = if (isFuture) MaterialTheme.colorScheme.primary else Color(0xFF7EC8A0),
    fontSize = 14.sp
)
```

## 七、归档 Snackbar 撤回机制

### 7.1 流程

| 步骤 | 触发 | 行为 |
|------|------|------|
| 1 | 用户左滑 → 抬手过 20% 阈值 | `viewModel.setExpandedDateId(date.id)` |
| 2 | 用户点归档按钮 | `viewModel.archiveDate(date.id)` → `SwipeableTodoBox` 自动收起 → Snackbar 显示 "已归档『<title>』"，actionLabel="撤回"，duration=3000ms |
| 3a | 3s 内点"撤回" | `viewModel.undoArchive()` → Snackbar 消失 → 卡片重新出现在原分组 |
| 3b | 3s 后无操作 | Snackbar 消失，`pendingArchive` 自动清空（实际已归档到 DB） |

### 7.2 互斥与边界

- **置顶卡归档**：归档时 `pinnedDateId` 同步清空（避免指向已归档数据）。
- **已归档卡重复点击归档按钮**：`repository.archive` 幂等（`UPDATE ... WHERE isArchived != archived` 可选加，但本次不强制要求）。
- **撤回时 DB 抛异常**：Snackbar 立即消失 + Toast 提示"恢复失败"；卡片不重新出现（DB 状态保持已归档）。

## 八、错误处理

| 场景 | 处理 |
|------|------|
| 归档时 DB 抛异常 | `viewModelScope.launch { try { repository.archive(id) } catch (e: Exception) { /* 走项目内统一的全局错误处理流程（具体入口后续由 host 注入），UI 弹简短 Toast */ } }` |
| 撤回时 DB 抛异常 | Snackbar 立即消失 + Toast 提示"恢复失败" |
| 物理删除级联失败 | `repository.delete()` 仍按原顺序调用 `relationDao` + `cardRelationRepository.removeAllForCard`；任一失败由 Repository 重新抛出，UI 弹 Toast |
| `formatDuration` 溢出 | 入参 `Long` 最大可表示约 2.92 亿年；`coerceAtLeast(0L)` 防止负数 |
| 搜索过滤空态 | `groupedDates` 为空 → `UnifiedEmptyState`（已存在） |
