# 置顶区跨区误判修复设计

> **文档创建日期**：2026-07-04
> **主题**：修复 `checkPinnedZoneCrossed` 把分隔按钮当作邻居导致置顶区内拖拽误翻转 `isPinned` 的 bug
> **状态**：待实施

## 一、问题背景

### 1.1 用户报告场景

10 条待办，名称 1-10，从上到下按序排列。1-4 是置顶状态，5-10 是待完成状态（Case A：置顶≥4）。

**操作**：将 4 拖拽到 1 上方释放。

**期望结果**：4, 1, 2, 3 都是置顶状态，顺序为 `[4, 1, 2, 3]`。

**实际结果**：1, 2, 3 仍是置顶，但 4 跳到了 3 和 5 之间（PendingDivider 之后），变成非置顶状态。

### 1.2 根因定位

**责任文件**：`app/src/main/java/com/corgimemo/app/ui/components/ReorderableLazyColumn.kt`

**责任函数**：`ReorderAlgorithms.checkPinnedZoneCrossed`（行 115-127）

**当前签名**：

```kotlin
fun checkPinnedZoneCrossed(
    displayItems: List<Boolean>,           // 调用方已 map { isPinned(it) }，丢失了类型
    draggedOriginalIsPinned: Boolean,
    draggedCurrentIndex: Int
): Boolean {
    if (draggedCurrentIndex < 0 || draggedCurrentIndex >= displayItems.size) return false
    val neighborIsPinned = when {
        draggedCurrentIndex > 0 -> displayItems[draggedCurrentIndex - 1]
        draggedCurrentIndex < displayItems.size - 1 -> displayItems[draggedCurrentIndex + 1]
        else -> draggedOriginalIsPinned
    }
    return draggedOriginalIsPinned != neighborIsPinned
}
```

**根因**：算法直接取 `displayItems[draggedCurrentIndex - 1]` 作为邻居，**没有跳过分隔按钮**（`PinnedDivider`/`PendingDivider`/`CompletedDivider`）。分隔按钮的 `isPinned=false`，被错误地当作"非置顶待办邻居"参与跨区判断。

### 1.3 场景重现（Case A，置顶≥4）

displayItems 结构（拖拽前）：

```
[PinnedDivider(0), P1(1), P2(2), P3(3), P4(4), PendingDivider(5), N1(6), ..., N6(11)]
isPinned: [false, true, true, true, true, false, false, ..., false]
```

**用户拖 P4 到 P1 上方**：

1. `onDragStarted`：`draggedOriginalIndex=4`，`draggedOriginalIsPinned=true`
2. 拖拽中库自动重排，P4 移到 index=1，displayItems 变为 `[PinnedDivider, P4, P1, P2, P3, PendingDivider, N1, ...]`
3. `onDragStopped`：`draggedCurrentIndex=1`
4. **算法执行**：
   - `displayItems[0]` = `PinnedDivider`，`isPinned=false`
   - `neighborIsPinned=false`
   - `crossed = (true != false) = true` ❌
5. `reorderOnDisplayList` 收到 `crossedPinnedZone=true`，进入 else 分支翻转 P4 的 `isPinned`
6. 重新分配 sortOrder 时按 `isPinned` 分组，P4(非置顶)被分到非置顶组，跳到 PendingDivider 之后

### 1.4 影响范围（两类 bug）

| 场景 | 期望 | 实际 | 根因 |
|------|------|------|------|
| 置顶项拖到置顶区内 | 不翻转 isPinned | 误翻为非置顶 | 邻居是 PinnedDivider(false)，误判跨区 |
| 非置顶项拖到置顶区内 | 翻转为置顶 | 未翻转 | 邻居是 PinnedDivider(false)，与 N1 相同，误判未跨区 |

## 二、修复方案

### 2.1 方案选择：方案 A（重构为泛型 + isPinned + isDivider 函数）

**对比过的方案**：

| 方案 | 签名 | 优点 | 缺点 |
|------|------|------|------|
| **A（采纳）** | 泛型 `<T>` + `isPinned` + `isDivider` 函数 | 类型安全、表达力强、调用方简洁 | 算法签名变化 |
| B | 算法不变，调用方预过滤 + 索引转换 | 算法不变 | 索引转换易 off-by-one（同类 bug 复发） |
| C | `List<Boolean>` + `isDivider: List<Boolean>` | 改动最小 | 两个 List 索引对齐是隐式契约 |

**采纳理由**：类型安全、表达力强、避免索引转换引入同类 bug。

### 2.2 新算法签名

```kotlin
fun <T> checkPinnedZoneCrossed(
    displayItems: List<T>,
    isPinned: (T) -> Boolean,
    isDivider: (T) -> Boolean,
    draggedOriginalIsPinned: Boolean,
    draggedCurrentIndex: Int
): Boolean
```

### 2.3 核心算法逻辑

替换"直接取相邻 index"为"向前/向后扫描跳过 divider，找第一个真实待办邻居"：

```kotlin
fun <T> checkPinnedZoneCrossed(
    displayItems: List<T>,
    isPinned: (T) -> Boolean,
    isDivider: (T) -> Boolean,
    draggedOriginalIsPinned: Boolean,
    draggedCurrentIndex: Int
): Boolean {
    if (draggedCurrentIndex < 0 || draggedCurrentIndex >= displayItems.size) return false

    // 优先向前找第一个非 divider 邻居
    var neighborIdx = -1
    for (i in draggedCurrentIndex - 1 downTo 0) {
        if (!isDivider(displayItems[i])) { neighborIdx = i; break }
    }
    // 前面没有则向后找
    if (neighborIdx < 0) {
        for (i in draggedCurrentIndex + 1 until displayItems.size) {
            if (!isDivider(displayItems[i])) { neighborIdx = i; break }
        }
    }
    // 整个列表都是 divider（理论上不可能）→ 不跨区
    if (neighborIdx < 0) return false

    return draggedOriginalIsPinned != isPinned(displayItems[neighborIdx])
}
```

### 2.4 修复效果验证

| 场景 | draggedCurrentIndex | 旧算法邻居 | 旧 crossed | 新算法邻居（跳过 divider） | 新 crossed | 正确性 |
|------|---------------------|------------|-----------|---------------------------|-----------|--------|
| P4→P1 上方（置顶区内） | 1 | PinnedDivider(false) | true ❌ | P1(true) | false ✓ | 不翻转，保持置顶 |
| N1→P1 上方（跨入置顶区） | 1 | PinnedDivider(false) | false ❌ | P1(true) | true ✓ | 翻转为置顶 |
| P1→P4 下方（置顶区内） | 4 | PendingDivider(false) | true ❌ | P4(true) | false ✓ | 不翻转 |
| P1→N1 上方（跨出置顶区） | 6 | N1(false) | true ✓ | N1(false) | true ✓ | 翻转为非置顶 |

## 四、边界情况处理

| 边界情况 | 处理策略 | 理由 |
|---------|----------|------|
| 整个列表只有 1 个待办 | `neighborIdx=-1` → 返回 false | 单项无法跨区 |
| 整个列表都是 divider（理论上不可能） | `neighborIdx=-1` → 返回 false | 防御性处理 |
| 拖到列表最顶部（toIndex=0，PinnedDivider 位置） | 向前找不到非 divider，向后找 P1 | 与现状一致，PinnedDivider 不可拖入由 `applyPinnedBoundary` 保证 |
| Case B（置顶<4，无 PinnedDivider） | 算法天然兼容，向前找到 PendingDivider 跳过，继续找到真实待办 | 算法不依赖 Case A/B 区分 |
| dividerIndex=-1（无已完成区） | 算法只关心 isPinned/isDivider，与 dividerIndex 无关 | 解耦 |

## 六、不变量保证

- **`HomeViewModel.reorderOnDisplayList` 逻辑不变**：仍接收 `crossedPinnedZone` 标志，仍用 `if (crossedPinnedZone) finalItem.copy(isPinned = !draggedItem.isPinned)` 翻转。修复只影响 `crossedPinnedZone` 的**计算正确性**，不影响其**消费方式**。
- **现有 7 个 `HomeViewModelReorderTest` 测试全部应继续通过**：因算法对无 divider 场景行为不变。
- **`HomeViewModel.mergeReorderOnDisplayList` 同步受益**：合并拖拽路径用同一算法，签名统一后两边都正确。

## 七、风险与回滚

### 7.1 风险评估

| 风险 | 等级 | 缓解 |
|------|------|------|
| 算法签名变化破坏其他调用方 | 低 | 全局搜索确认只有 2 处调用（单拖/合并拖） |
| `isDivider` lambda 在 HomeScreen 中漏写 | 低 | 编译期检查（参数非可选） |
| 新算法在 Case B 行为变化 | 低 | Case B 也会跳过 PendingDivider，行为更正确 |

### 7.2 回滚方案

若发现问题，回滚只需还原 `checkPinnedZoneCrossed` 签名和两处调用点。改动集中在 3 个文件，回滚成本低。
