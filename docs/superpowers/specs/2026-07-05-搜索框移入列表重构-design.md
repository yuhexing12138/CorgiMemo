# 搜索框移入列表重构 设计文档

## 背景与问题

待办页搜索框在添加置顶按钮功能后，显隐过渡过程存在明显的跳跃感。尽管 2026-07-05 已实施"两阶段架构"修复（commits `ac26c4e`、`0c99f34`），跳跃感仍然存在。

### 当前架构根因

[HomeScreen.kt#L533-L680](file:///c:/Users/EDY/Desktop/CorgiMemo/app/src/main/java/com/corgimemo/app/ui/screens/home/HomeScreen.kt#L533-L680) 的状态链路：

```
LazyListState.firstVisibleItemScrollOffset（每帧变化）
    ↓ derivedStateOf
scrollDrivenProgress
    ↓ if (isScrolling) scrollDrivenProgress else animatableProgress.value
currentProgress
    ↓ layout 修饰符动态改高度
SearchBox 高度变化
    ↓ Column 重新分配空间
LazyColumn 视口高度变化 → firstVisibleItemScrollOffset 反馈变化
```

| # | 问题 | 影响 |
|---|------|------|
| 1 | **`isScrolling` 切换时的 1-2 帧跳跃** | 手指松开瞬间 `isScrolling` 变 false，`currentProgress` 切换到 `animatableProgress.value`（旧值），`LaunchedEffect` 的 `snapTo` 要下一帧才执行，造成 1-2 帧视觉跳跃 |
| 2 | **布局反馈循环** | SearchBox 通过 `layout` 修饰符改高度 → LazyColumn 视口变化 → `firstVisibleItemScrollOffset` 变化 → `scrollDrivenProgress` 变化 → SearchBox 高度再变化 |
| 3 | **置顶按钮放大反馈** | 置顶按钮使首项变短（PinnedDivider），视口变化对滚动位置的相对影响更大，放大反馈循环 |
| 4 | **`Animatable.snapTo` 在 effect 协程中执行** | `snapTo` 本身同步，但被 `LaunchedEffect` 包裹后变成异步，快速滚动时累积 |

**本质**：搜索框作为 LazyColumn 外的折叠头部，与 LazyColumn 视口存在耦合，无法彻底消除反馈循环。

## 设计目标

1. **彻底消除跳跃感**：搜索框显隐完全交给 LazyColumn 原生滚动，无中转状态
2. **保留下拉刷新柯基位置**：柯基动画仍位于搜索框与待办卡片区之间
3. **保留拖拽排序**：搜索框不参与拖拽，待办卡片排序不受影响
4. **减少代码复杂度**：删除 `scrollDrivenProgress`/`animatableProgress`/`computeScrollDrivenProgress` 等中转状态

## 架构设计

### 新布局结构

```
Column (fillMaxSize)
├─ SolarTermCard (AnimatedVisibility，保持不变，在 LazyColumn 之外)
└─ Box (fillMaxSize, weight=1f, nestedScroll=pullRefreshConnection)
    └─ ZonedReorderableLazyColumn (fillMaxSize)
        ├─ item { SearchBar(...) }                          ← 第 0 项，随滚动自然移动
        ├─ item { PullRefreshSpacer(pullOffset, corgi) }    ← 第 1 项，高度=pullOffset，含柯基
        └─ itemsIndexed(displayItems) { ... }               ← 第 2 项起，待办卡片
```

### 关键变化

| 组件 | 旧位置 | 新位置 | 行为变化 |
|------|--------|--------|---------|
| SearchBar | LazyColumn 外，`layout` 修饰符动态改高度 | LazyColumn 第 0 项 | 随滚动自然出屏，无折叠动画 |
| 下拉刷新柯基 | 外层 Box 的 overlay | LazyColumn 第 1 项（spacer） | 高度随 `pullOffset` 增长，柯基居中 |
| `translationY = pullOffset` | 外层 Box | **删除** | 由 spacer item 高度变化替代 |
| `scrollDrivenProgress` | derivedStateOf | **删除** | 不再需要 |
| `animatableProgress` | Animatable | **删除** | 不再需要 |
| `currentProgress` | if 切换 | **删除** | 不再需要 |
| 级联偏移 `graphicsLayer` | LazyColumn | **删除** | 不再需要 |
| `computeScrollDrivenProgress()` | 顶层函数 | **删除** | 不再需要 |

### 数据流

```
用户滚动 → LazyListState 自然滚动 → SearchBar 随之出屏/入屏（无中转状态）

用户下拉刷新（在顶部）→ NestedScrollConnection 拦截 → pullOffset 增长
                     → spacer item 高度增长 → 柯基显示 → 卡片被推下
                     → 释放 → onRefresh() → pullOffset 回弹到 0
```

### 核心收益

1. **彻底消除布局反馈循环**：SearchBar 不再通过 `layout` 修饰符改变高度，LazyColumn 视口恒定
2. **彻底消除 1-2 帧切换跳跃**：不再有 `isScrolling` 切换 + `snapTo` 延迟
3. **下拉刷新柯基天然就位**：spacer 在 SearchBar 和卡片之间，无需 overlay 定位

### PullRefreshSpacer 组件

将原 [HomeScreen.kt#L809-L815](file:///c:/Users/EDY/Desktop/CorgiMemo/app/src/main/java/com/corgimemo/app/ui/screens/home/HomeScreen.kt#L809-L815) 的 `CorgiPullRefreshIndicator` 调用封装为 spacer：

```kotlin
/**
 * 下拉刷新空白区 + 柯基动画
 *
 * 作为 LazyColumn 的前置 item，高度随 pullOffset 增长，
 * 替代原外层 Box 的 translationY 偏移方案。
 *
 * @param pullOffset 当前下拉偏移量（px）
 * @param state 下拉刷新状态
 * @param maxPullHeightPx 最大下拉高度（px）
 * @param refreshThresholdPx 刷新触发阈值（px）
 * @param modifier Modifier
 */
@Composable
private fun PullRefreshSpacer(
    pullOffset: Float,
    state: PullRefreshState,
    maxPullHeightPx: Float,
    refreshThresholdPx: Float,
    modifier: Modifier = Modifier
) {
    // 高度 = pullOffset（px → dp），未下拉时高度为 0 不可见
    val heightDp = with(LocalDensity.current) { pullOffset.toDp() }
    Box(
        modifier = modifier
            .height(heightDp)
            .clipToBounds()
    ) {
        CorgiPullRefreshIndicator(
            pullOffset = pullOffset,
            state = state,
            maxPullHeightPx = maxPullHeightPx,
            refreshThresholdPx = refreshThresholdPx,
            modifier = Modifier.fillMaxWidth()
        )
    }
}
```

## 边界情况处理

| 场景 | 处理方式 |
|------|---------|
| **搜索框已滚出屏幕时下拉** | `NestedScrollConnection` 仅在 `firstVisibleItemIndex == 0` 时拦截，搜索框滚出后下拉为普通滚动 |
| **搜索词活跃时滚动** | SearchBar 随列表滚出，过滤仍生效；用户滚回顶部可访问 SearchBar 清空搜索词 |
| **搜索词活跃时下拉刷新** | 列表在顶部，SearchBar 可见，下拉触发刷新，过滤结果刷新 |
| **拖拽排序时** | `headerContent` 的 item 不包裹 `ReorderableItem`，不参与拖拽；`onMove` 索引偏移 `headerItemCount=2` |
| **批量模式时** | `isDragEnabled=false`，SearchBar 仍可正常输入（与原行为一致） |
| **SolarTermCard 显示时** | SolarTermCard 在 LazyColumn 之外，挤压 LazyColumn 可用高度，不影响 SearchBar 作为首项 |
| **列表为空时** | displayItems 仅含 divider，SearchBar 仍为首项可见 |
| **快速 fling 滚动** | LazyColumn 原生 fling，SearchBar 自然滚出/入，无 `isScrolling` 切换跳跃 |

## 下拉刷新时序验证

**正常下拉刷新流程**（列表在顶部）：

```
初始状态: SearchBar(可见) + Spacer(高=0) + 待办卡片
   ↓ 用户手指下滑（在列表顶部）
NestedScrollConnection.onPreScroll 拦截 → pullOffset 增长
   ↓
Spacer 高度增长 → 柯基动画显示 → 待办卡片被推下
   ↓ 释放
onRefresh() 触发 → pullOffset 回弹到 0 → Spacer 高度归 0
   ↓
刷新完成 → isRefreshing=false → onRefreshComplete() 收尾
```

**关键验证点**：
- 柯基位于 SearchBar 下方、待办卡片上方（满足用户约束）
- SearchBar 在下拉过程中保持原位（LazyColumn 滚动位置仍为 0）
- 待办卡片随 spacer 增长被推下（视觉与原 `translationY` 一致）

### `isRefreshing` 回弹处理

[HomeScreen.kt#L730-L732](file:///c:/Users/EDY/Desktop/CorgiMemo/app/src/main/java/com/corgimemo/app/ui/screens/home/HomeScreen.kt#L730-L732) 的 `LaunchedEffect(isRefreshing)` 保持不变：

```kotlin

    if (!isRefreshing) pullRefreshState.onRefreshComplete()
}
```

`onRefreshComplete()` 内部会将 `pullOffset` 动画回弹到 0，spacer 高度随之归 0。此逻辑无需改动。

### `nestedScrollConnection` 行为确认

`PullRefreshStateHolder`（[PullRefreshStateHolder.kt](file:///c:/Users/EDY/Desktop/CorgiMemo/app/src/main/java/com/corgimemo/app/ui/components/PullRefreshStateHolder.kt)）使用 `onPostScroll` 模式，**不需要持有 `lazyListState` 引用**：

```kotlin
// PullRefreshStateHolder.kt 实际实现（无需改动）
override fun onPostScroll(consumed: Offset, available: Offset, source: NestedScrollSource): Offset {
    if (state == PullRefreshState.REFRESHING) return Offset.Zero
    // LazyColumn 在顶部无法继续向下滚动时，剩余 delta 转为 pullOffset
    if (available.y > 0f) {
        rawPullOffset += available.y
        val newOffset = computeDampedOffset(rawPullOffset, maxPullHeightPx)
        val consumedY = newOffset - pullOffset
        pullOffset = newOffset
        updateState(isReleased = false)
        return Offset(0f, consumedY)
    }
    return Offset.Zero
}
```

**原理**：`onPostScroll` 在 LazyColumn 消费完滚动后调用。LazyColumn 在顶部无法向下滚动时，剩余 `available.y > 0` 自动流向 `onPostScroll`，转为 `pullOffset`。无需手动检查 `firstVisibleItemIndex`。

**`onPreScroll` 收回逻辑**：当 `pullOffset > 0` 且用户上推（`available.y < 0`）时，父组件优先消费 delta 收回空白，再交给列表滚动。此逻辑同样不依赖 `lazyListState`。

**结论**：`PullRefreshStateHolder` 无需改动，直接复用现有实现。

## 风险与缓解

| 风险 | 缓解 |
|------|------|
| `headerItemCount` 与实际 header item 数量不一致 → 索引错乱 | 调用方需保证一致；可加 `assert(headerItemCount >= 0)` |
| header item 的 key 未传给 `rememberReorderableLazyListState` | header 不包裹 `ReorderableItem`，不参与拖拽，无需 key |
| `onMove` 中 `to.index` 落在 header 范围（< headerItemCount） | `from.index - headerItemCount` 校验为 Todo 后，`to` 落在 header 会被库过滤（header 不在 reorderableKeys） |
| 搜索框滚出后用户无法直接清空搜索词 | 用户可滚回顶部清空；UX 已确认可接受（自然滚动出屏） |
| spacer 高度变化导致 LazyColumn 内容重排 | LazyColumn 在 scroll position=0 时，spacer 增长仅推动后续 item 下移，不影响 scroll position |
