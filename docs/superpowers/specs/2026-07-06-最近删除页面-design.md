## 一、问题背景

### 现象

用户在待办页（TODO Tab）打开侧边栏（抽屉），点击「最近删除」入口后，**没有任何页面跳转或 UI 变化**，仅抽屉自动关闭。

### 根因分析

| 序号 | 缺失点 | 影响 |
|------|--------|------|
| 1 | `Screen.kt` 中无 `RecentlyDeleted` 路由定义 | 无路由可 navigate |
| 2 | `AppNavHost.kt` 中无 `composable("recently_deleted")` 注册 | 即使有路由也找不到目标 |
| 3 | 无 `RecentlyDeletedScreen.kt` 屏幕文件 | 无目标页面渲染 |
| 4 | 无 `RecentlyDeletedViewModel` | 无业务逻辑承载层 |
| 5 | `MainScreen.kt:265-267` `onRecentlyDeletedClick` 仅关闭抽屉，无 `navController.navigate(...)` | 点击后无导航动作 |

### 数据层现状（已就绪）

### 设计目标

- 修复"点击最近删除无反应"问题
- 提供完整的最近删除管理界面（恢复/永久删除/清空/显示时间）
- 按时间分组、相对时间显示，符合用户心理预期
- 自动化 30 天清理，避免数据库无限增长
- 单一职责：新建独立 ViewModel + Screen，与 HomeViewModel 解耦

### mapToUiState 算法

```kotlin
private fun mapToUiState(
    deleted: List<DeletedTodo>,
    categories: List<Category>
): RecentlyDeletedUiState {
    val now = System.currentTimeMillis()
    val categoryMap = categories.associateBy { it.id }

    // 1. 投影为 ListItem
    val items = deleted.map { d ->
        DeletedTodoListItem(
            id = d.id,
            title = d.title,
            originalCategoryId = d.categoryId,
            categoryName = d.categoryId?.let { categoryMap[it]?.name },
            deletedAt = d.deletedAt,
            relativeTime = TimeClassifier.formatRelativeTime(d.deletedAt, now)
        )
    }

    // 2. 按时间分组（组内按 deletedAt 倒序）
    val grouped = items
        .sortedByDescending { it.deletedAt }
        .groupBy { TimeClassifier.classifyByTime(it.deletedAt, now) }
        .map { (kind, list) ->
            DeletedTodoGroup(
                kind = kind,
                title = when (kind) {
                    TODAY -> "今天"
                    YESTERDAY -> "昨天"
                    THIS_WEEK -> "本周"
                    EARLIER -> "更早"
                },
                items = list
            )
        }

    return RecentlyDeletedUiState(
        groups = grouped,
        totalCount = items.size,
        showClearAllDialog = _uiState.value.showClearAllDialog,
        isLoading = false
    )
}
```

### restoreTodo 关键逻辑

```kotlin
fun restoreTodo(deletedTodoId: Long) = viewModelScope.launch {
    runCatching {
        val deleted = deletedTodoRepository.getByIdBlocking(deletedTodoId) ?: run {
            _events.emit(UiEvent.ShowSnackBar("该待办不存在"))
            return@launch
        }
        val todo = DeletedTodo.toTodoItem(deleted)

        // 1. 决定 categoryId（原分组存在则保留，否则 null）
        val finalCategoryId = todo.categoryId?.let { id ->
            if (categoryRepository.getByIdBlocking(id) != null) id else null
        }

        // 2. 决定 sortOrder（PENDING 区最大值 + 1）
        val maxSortOrder = todoRepository.getMaxSortOrderInZone(
            zone = TodoZone.PENDING,
            isPinned = false
        ) ?: 9999  // PENDING 起始值 - 1

        val restoredTodo = todo.copy(
            categoryId = finalCategoryId,
            sortOrder = maxSortOrder + 1,
            isPinned = false,
            status = 0  // PENDING
        )

        // 3. 写入 + 清理 deleted_todo
        todoRepository.insertTodo(restoredTodo)
        deletedTodoRepository.permanentlyDelete(deletedTodoId)
        _events.emit(UiEvent.ShowSnackBar("已恢复"))
    }.onFailure {
        _events.emit(UiEvent.ShowSnackBar("恢复失败，请重试"))
    }
}
```

> **依赖补充**：TodoRepository 当前未提供 `getMaxSortOrderInZone` / `getByIdBlocking` 方法。本任务需在 `TodoRepository` 中新增这两个方法（均为 IO 调度、阻塞获取）。

### SnackBar 撤销设计

- 永久删除单条后，缓存被删除的 DeletedTodo 临时数据到 ViewModel 的 `_lastDeletedForUndo`
- SnackBar 出现 "已永久删除 [撤销]" 按钮
- 5 秒内点击 [撤销] → 调用 `deletedTodoRepository.insertDeletedTodo(cached)`
- 5 秒后 → 临时数据丢弃
- 清空全部：不提供 SnackBar 撤销（避免歧义）

## 七、错误处理

| 场景 | 错误处理 |
|------|----------|
| `getAllDeletedTodos()` Flow 异常 | emit 空列表 + `_uiState.copy(isLoading=false)` |
| `restoreTodo` 时 `getById` 返回 null | SnackBar "该待办不存在" |
| `restoreTodo` 时 `insertTodo` 失败 | SnackBar "恢复失败，请重试" |
| `permanentlyDelete` 失败 | SnackBar "删除失败，请重试" |
| `permanentlyDeleteAll` 失败 | AlertDialog 关闭 + SnackBar "清空失败，请重试" |
| `cleanUpOldDeletedTodos` 失败 | 静默忽略（下次启动重试） |

**实现方式**：ViewModel 内 `runCatching { ... }.onFailure { ... }` 捕获；CoroutineExceptionHandler 不在 ViewModel 设置（避免吞掉 crash）。
