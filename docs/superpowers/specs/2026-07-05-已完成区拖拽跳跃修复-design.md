## 1. 背景与问题

### 1.1 用户报告的场景

已知 10 条待办，名称为 1-10，从上到下按序排列。其中 **7, 8, 9, 10 是已完成状态**，其他都是待完成状态。当用户将 7 拖拽到 8 和 9 之间释放时，**7 发生了位置跳跃**：

- 实际结果：8, 9, 7（错误）
- 期望结果：8, 7, 9

### 1.2 影响

已完成区内的拖拽排序无法正常使用，拖拽释放后位置不可预期。

---

## 2. 根因分析

### 2.1 触发链路

```
用户拖拽 7 到 8 和 9 之间释放
  ↓
ReorderableLazyColumn.onDragStopped 回调
  ↓
displayItems.indexOfFirst { key(it) == key(item) } 找到 7 的当前位置（index=2）
  ↓
ReorderAlgorithms.checkPinnedZoneCrossed 判定是否跨置顶区
  ↓
算法返回 true（误判）→ crossedPinnedZone = true
  ↓
HomeViewModel.reorderOnDisplayList 把 7 的 isPinned 翻转（false → true）
  ↓
持久化后，visibleCompletedTodos 排序把置顶项排到非置顶项之前
  ↓
位置跳跃
```

### 2.2 算法层根因

`checkPinnedZoneCrossed` 算法（[ReorderableLazyColumn.kt:138-178](file:///c:/Users/EDY/Desktop/CorgiMemo/app/src/main/java/com/corgimemo/app/ui/components/ReorderableLazyColumn.kt#L138-178)）当前的判定逻辑：

```kotlin
// 1. 扫描前面最近的 divider，确定 currentZone
// 2. 基于 divider 类型直接判定：
val originalZone = if (draggedOriginalIsPinned) DividerKind.PINNED else DividerKind.PENDING
return currentZone != originalZone
```

**Bug 本质**：

| 项 | 当前算法 | 正确行为 |
|----|----------|----------|
| `currentZone` | `COMPLETED`（前面最近 divider 是 CompletedDivider） | `COMPLETED`（正确） |
| `originalZone` | `PENDING`（仅依据 `draggedOriginalIsPinned=false`） | 不应参与比较 |
| 比较结果 | `PENDING != COMPLETED` → 返回 `true`（误判） | 应返回 `false` |
| 副作用 | 翻转 `isPinned` | 不应翻转 |

### 2.3 设计错误

`originalZone` 的推导基于"已完成项的 isPinned 必为 false"的隐式假设，但这并不正确：

1. **已完成区同样支持置顶**（可见 `visibleCompletedTodos` 的排序：`compareByDescending<TodoItem> { it.isPinned }`）
2. 已完成区内没有 `PinnedDivider`，**不能用 divider 类型直接判断置顶边界**
3. `COMPLETED` 区域的语义是"已完成区"，与 `PINNED/PENDING` 是不同维度，不能直接比较

### 2.4 测试验证

已在 [ReorderAlgorithmsTest.kt](file:///c:/Users/EDY/Desktop/CorgiMemo/app/src/test/java/com/corgimemo/app/ui/components/ReorderAlgorithmsTest.kt) 新增 3 个测试用例验证根因，运行结果**全部失败**（与预期一致）：

| 测试名 | 场景 | 当前结果 | 期望结果 |
|--------|------|----------|----------|
| `已完成区内向前拖拽不应跨区` | 拖 7 到 8 和 9 之间 | 失败（返回 true） | 通过（返回 false） |
| `已完成区内向后拖拽不应跨区` | 拖 7 到 9 和 10 之间 | 失败（返回 true） | 通过（返回 false） |
| `已完成区内拖到开头不应跨区` | 拖 7 到 CompletedDivider 后 | 失败（返回 true） | 通过（返回 false） |

## 3. 设计方案

### 3.1 方案选型

| 方案 | 描述 | 优点 | 缺点 |
|------|------|------|------|
| **方案 A（采用）** | 修改 `checkPinnedZoneCrossed`：`currentZone=COMPLETED` 时回退到"找邻居 + 比较 isPinned" | 从根因修复，符合 divider 语义 | 需注意已完成区内置顶切换的边界 |
| 方案 B | 在 `reorderOnDisplayList` 业务层忽略已完成区的 `crossedPinnedZone` | 改动小 | 治标不治本，算法层 bug 仍存在 |
| 方案 C | 通过 `draggedOriginalIndex` 扫描原位置 divider 推导 `originalZone` | 更精准 | 接口变更较大，需传入额外参数 |

### 3.2 选型理由

选择方案 A 的核心理由：

1. **职责匹配**：`checkPinnedZoneCrossed` 的职责是检测"置顶区跨越"。已完成区内没有 `PinnedDivider`，无法用 divider 类型判断置顶边界。
2. **语义清晰**：`COMPLETED` 区域应与"无 divider 场景"采用相同的回退逻辑（找邻居 + 比较 isPinned）。
3. **最小改动**：只需修改算法一个分支，不影响现有 PINNED/PENDING 区域逻辑。
4. **向后兼容**：已完成区内的同区域拖拽本来就应返回 `false`（与置顶区/待完成区一致）。

### 3.3 算法修改

#### 修改前的核心逻辑（[ReorderableLazyColumn.kt:147-161](file:///c:/Users/EDY/Desktop/CorgiMemo/app/src/main/java/com/corgimemo/app/ui/components/ReorderableLazyColumn.kt#L147-161)）

```kotlin
// 1. 扫描被拖项前面最近的 divider，确定其当前所在区域
var currentZone: DividerKind? = null
for (i in draggedCurrentIndex - 1 downTo 0) {
    val kind = dividerKind(displayItems[i])
    if (kind != null) {
        currentZone = kind
        break
    }
}

// 2. 若找到 divider，基于 divider 类型直接判定
if (currentZone != null) {
    val originalZone = if (draggedOriginalIsPinned) DividerKind.PINNED else DividerKind.PENDING
    return currentZone != originalZone
}

// 3. 无 divider → 回退到"找邻居 + 比较 isPinned"
// ...
```

#### 修改后的核心逻辑

```kotlin
// 1. 扫描被拖项前面最近的 divider，确定其当前所在区域
var currentZone: DividerKind? = null
for (i in draggedCurrentIndex - 1 downTo 0) {
    val kind = dividerKind(displayItems[i])
    if (kind != null) {
        currentZone = kind
        break
    }
}

// 2. 基于 divider 类型判定（仅对 PINNED 和 PENDING 区有效）
//    已完成区（COMPLETED）没有 PinnedDivider，无法用 divider 判断置顶边界，
//    应回退到"找邻居 + 比较 isPinned"逻辑
if (currentZone == DividerKind.PINNED || currentZone == DividerKind.PENDING) {
    val originalZone = if (draggedOriginalIsPinned) DividerKind.PINNED else DividerKind.PENDING
    return currentZone != originalZone
}

// 3. 无 divider 或 currentZone=COMPLETED → 回退到"找邻居 + 比较 isPinned"
//    （保持向后兼容，与无 divider 场景一致）
var neighborIdx = -1
val prevIdx = draggedCurrentIndex - 1
if (prevIdx >= 0 && dividerKind(displayItems[prevIdx]) == null) {
    neighborIdx = prevIdx
}
if (neighborIdx < 0) {
    val nextIdx = draggedCurrentIndex + 1
    if (nextIdx < displayItems.size && dividerKind(displayItems[nextIdx]) == null) {
        neighborIdx = nextIdx
    }
}
if (neighborIdx < 0) return false

return draggedOriginalIsPinned != isPinned(displayItems[neighborIdx])
```

### 3.4 关键不变式

修改后的算法保持以下不变式：

1. **置顶区跨区检测**：`PINNED` ↔ `PENDING` 跨越 PinnedDivider 时仍正确返回 `true`
2. **同区域内拖拽不跨区**：置顶区内、待完成区内、已完成区内同区域拖拽均返回 `false`
3. **divider 是区域边界**：找邻居时遇 divider 即停止，不跨越
4. **已完成区内置顶切换**：通过"找邻居 + 比较 isPinned"逻辑判定（与原算法在无 divider 场景的行为一致）

## 6. 风险与缓解

### 6.1 已完成区内置顶切换

**风险**：方案 A 在已完成区内回退到"找邻居 + 比较 isPinned"。如果用户拖动一个已完成非置顶项到已完成置顶项之前，期望它变成置顶，是否能正确触发？

**缓解**：算法第 3 步"找邻居 + 比较 isPinned"会找到已完成区内的置顶邻居，与 `draggedOriginalIsPinned=false` 比较后返回 `true`，触发翻转。行为与原算法在无 divider 场景下一致。

### 6.2 dividerIndex = -1 场景

**风险**：无已完成区分隔按钮的场景下，`currentZone = null`，进入回退逻辑。

**缓解**：现有 `dividerIndex 负一同区拖拽应持久化` 测试已覆盖此场景，修复不影响该路径。
