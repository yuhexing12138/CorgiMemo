# 待办卡片拖拽排序重构设计

- 日期：2026-06-29
- 范围：待办列表长按拖拽排序的彻底重构
- 关联文件：`app/src/main/java/com/corgimemo/app/ui/components/ReorderableLazyColumn.kt`
- 本地库源码：`Reorderable/`（Calvin-LL/Reorderable，commit `2eb3b59a`）

## 一、背景与问题

待办事项应用的长按拖拽排序存在两个严重交互异常：

| 编号 | 现象 | 触发条件 |
|---|---|---|
| 问题1 | 长按列表第一个待办卡片向下拖动时，整个应用页面触发不受控制的快速向下滑动，界面失去响应 | 长按首项 + 向下拖动 |
| 问题2 | 长按非首项卡片与第一个卡片交换位置时，被拖卡片突然向上移出可视区域，操作中断 | 长按其他项 + 与首项交换 |

此前已有多轮局部修复（见 `2026-06-29-todo-drag-firstcard-bounds-design.md`、`2026-06-29-todo-drag-release-jitter-fix-design.md`、`2026-06-28-拖拽高度差跳动bug修复-design.md`），但根因未除，问题反复。本次决定彻底重构。

## 二、根因分析

当前实现位于 `ReorderableLazyColumn.kt`，采用自研 `pointerInput { awaitEachGesture { ... } }` 手动状态机，维护 15+ 个状态变量。核心缺陷是**三套坐标系混用**：

| 状态变量 | 坐标系 | 用途 |
|---|---|---|
| `draggedListCenterY` | 逻辑坐标（`targetIndex × averageItemHeightPx`） | 被拖项应有中心 |
| `fingerY` | 视口坐标 | 手指位置 |
| `scrollCompensationY` | 滚动累积量 | auto-scroll 补偿 |

`dragOffsetY = fingerY - draggedListCenterY - scrollCompensationY` 把三者相减，坐标系不匹配导致计算结果错误。

### 问题1根因
auto-scroll 用 `fingerY`（视口坐标）判断方向（`ReorderableLazyColumn.kt:580-588`）：
```kotlin
fingerY < edgeThresholdPx -> 正值 scrollBy   // 触发向上滚动
```
被拖项在列表顶部时，手指 Y 天然偏小（落入顶部边缘阈值），即使向下拖动也误触发"向上滚动"。叠加 `scrollCompensationY` 补偿与 `draggedListCenterY` 不随之调整，形成正反馈失控。

### 问题2根因
被拖项从索引 N 交换到索引 0 时（`ReorderableLazyColumn.kt:759-763`）：
```
draggedListCenterY = computeDraggedListCenterY(0, size, avgH) = 0 × avgH + size/2 ≈ size/2（很小）
dragOffsetY = fingerY(视口，大) - draggedListCenterY(小) - scrollCompensationY
```
当 `scrollCompensationY` 累积为正（auto-scroll 向上滚动）时，`dragOffsetY` 算出大负值，卡片向上飞出视口。

### 设计层面的根本缺陷
- 坐标系不统一是结构性问题，局部修补无法根除
- 手动状态机复杂（释放期 `isReleasing` hack、反向交换锁定 `lastSwapTargetKey`、`dragScrollBlocker` 拦截等），脆弱难维护
- 自研无法支持首可见项动画（注释承认"第一可见项不动画"是已知问题）

### 4.3 核心状态与坐标系统

`ReorderableLazyCollectionState`（`ReorderableLazyCollection.kt:257-678`）内部状态：

| 状态 | 类型 | 说明 |
|---|---|---|
| `draggingItemKey` | `Any?` | 被拖项 key |
| `draggingItemDraggedDelta` | `Offset` | 累计拖拽增量 |
| `draggingItemInitialOffset` | `IntOffset` | 拖拽开始时被拖项的 offset |
| `draggingItemLayoutInfo` | `LazyCollectionItemInfo?` | 从 `visibleItemsInfo` 实时查找 |
| `oldDraggingItemIndex` | `Int?` | 移动前的索引（防跳变） |
| `predictedDraggingItemOffset` | `IntOffset?` | 移动后预测位置（防一帧跳变） |
| `previousDraggingItemKey` | `Any?` | 释放后回弹动画的项 key |
| `previousDraggingItemOffset` | `Animatable<Offset>` | 释放回弹动画 |

**坐标计算核心**（`ReorderableLazyCollection.kt:316-331`）：
```kotlin
internal val draggingItemOffset: Offset
    get() = (draggingItemLayoutInfo?.let {
        val offset = if (it.index != oldDraggingItemIndex || oldDraggingItemIndex == null) {
            oldDraggingItemIndex = null
            predictedDraggingItemOffset = null
            it.offset                                  // 正常：用当前 layoutInfo 的 offset
        } else {
            predictedDraggingItemOffset ?: it.offset    // 移动中：用预测值防跳变
        }
        draggingItemDraggedDelta + (draggingItemInitialOffset.toOffset() - offset.toOffset())
            .reverseAxisIfNecessary()
    }) ?: Offset.Zero
```

**统一坐标系**：所有坐标均来自 `LazyListItemInfo.offset`（视口坐标），`draggingItemDraggedDelta` 是手指增量。不存在逻辑坐标与视口坐标混用。这从根本上解决问题2。

### 4.4 自动滚动机制

`onDrag` 方法（`ReorderableLazyCollection.kt:417-501`）核心逻辑：

```
1. draggingItemDraggedDelta += offset（累加手指增量）
2. 计算 handleOffset = 被拖项 startOffset + draggingItemHandleOffset + beforeContentPadding
   （handleOffset = 手柄中心相对内容区起点的位置）
3. distanceFromStart = handleOffset - contentStartOffset
   distanceFromEnd = contentEndOffset - handleOffset
4. 若 distanceFromStart < scrollThreshold(48dp)：
     scroller.start(BACKWARD, speedMultiplier, maxScrollDistance, onScroll)
   若 distanceFromEnd < scrollThreshold：
     scroller.start(FORWARD, speedMultiplier, maxScrollDistance, onScroll)
   否则：scroller.tryStop()
5. 若未在滚动：查找目标项并 moveItems
```

**速度倍数**（`getScrollSpeedMultiplier`，`:671-677`）：
```
speedMultiplier = (1 - (distance + threshold) / (threshold * 2)).coerceIn(0,1) * 10
```
距离越近倍数越大（1..10），实现变速滚动。

**关键差异**：库基于 `handleOffset`（手柄中心，跟随被拖项）判断方向，而非 `fingerY`。首项向下拖时被拖项整体下移，handleOffset 增大、远离顶部边缘，不误触发向上滚动。这从根本上解决问题1。

### 4.5 移动判定

LazyList 垂直方向的 `shouldItemMove`（`ReorderableLazyList.kt:177-185`）：
```kotlin
shouldItemMove = { draggingItem, item ->
    item.center.y in draggingItem.top..<draggingItem.bottom
}
```
即：目标项中心 Y 落在被拖项矩形上下边界之间时触发移动。基于视口坐标，无坐标系换算。

### 4.6 释放回弹动画（内置）

`onDragStop`（`ReorderableLazyCollection.kt:391-415`）：
```kotlin
previousDraggingItemKey = draggingItemKey
previousDraggingItemOffset.snapTo(draggingItemOffset)           // 从当前位置开始
previousDraggingItemOffset.animateTo(
    Offset.Zero,
    spring(stiffness = Spring.StiffnessMediumLow,                // 弹簧动画
           visibilityThreshold = Offset.VisibilityThreshold)
)
previousDraggingItemKey = null
```
`ReorderableItem` 中对 `previousDraggingItemKey` 的项应用 `graphicsLayer.translationY = previousDraggingItemOffset.value.y`，实现松手后平滑归位。无需自研释放动画。

### 4.7 首可见项保护

`moveItems`（`ReorderableLazyCollection.kt:619-663`）中：
```kotlin
if (draggingItem.index == state.firstVisibleItemIndex ||
    targetItem.index == state.firstVisibleItemIndex) {
    state.requestScrollToItem(
        state.firstVisibleItemIndex,
        state.firstVisibleItemScrollOffset
    )
}
```
交换涉及首可见项时，调用 `requestScrollToItem` 保持首项位置不变，防止跳跃。这解决了自研实现"第一可见项不动画"的难题。

### 4.8 ReorderableColumn（非懒列表）

源码 `ReorderableList.kt:376-418`：
```kotlin
@Composable
fun <T> ReorderableColumn(
    list: List<T>,
    onSettle: (fromIndex: Int, toIndex: Int) -> Unit,    // 松手时调用
    modifier: Modifier = Modifier,
    verticalArrangement: Arrangement.Vertical = Arrangement.Top,
    horizontalAlignment: Alignment.Horizontal = Alignment.Start,
    onMove: () -> Unit = {},                              // 拖拽中移动时调用（可选）
    content: @Composable ReorderableColumnScope.(index: Int, item: T, isDragging: Boolean) -> Unit,
)
```

**与项目当前签名的差异**：
- 参数名 `list`（项目用 `items`）
- 回调名 `onSettle`（项目用 `onReorder`）
- content scope 是 `ReorderableColumnScope`（继承 `ColumnScope`，项目用裸 lambda）
- content 参数 `(index, item, isDragging)`（项目一致）

重写时需在项目 `ReorderableColumn` 内部适配：`items` → 库的 `list`，`onReorder` → 库的 `onSettle`，content 包装为 `ReorderableColumnScope` 调用。

**注意**：`ReorderableColumn` 无自动滚动（非懒列表，全部渲染），与现有 Column 场景无边缘滚动需求一致。

### 对外 API（保持不变，HomeScreen / InspirationEditScreen 零改动）

```kotlin
// ReorderableColumn - 签名不变
@Composable
fun <T> ReorderableColumn(
    items: List<T>,
    onReorder: (fromIndex: Int, toIndex: Int) -> Unit,
    modifier: Modifier = Modifier,
    content: @Composable (index: Int, item: T, isDragging: Boolean) -> Unit
)
```

### 内部状态（从 15+ 精简到 4 个）

```kotlin
var displayItems by remember { mutableStateOf(items) }        // 显示列表（拖拽中可重排）
var isDragActive by remember { mutableStateOf(false) }        // 拖拽激活（协调子项/左滑）
var draggedOriginalIndex by remember { mutableIntStateOf(-1) } // 被拖项原始索引（提交时用）
var crossedPinnedZone by remember { mutableStateOf(false) }   // 是否跨越置顶区
```

### ReorderableLazyColumn 状态流

```
rememberReorderableLazyListState(lazyListState, onMove = { from, to ->
    ① 校验 from.index/to.index 对应 displayItems 是否可拖拽（isDraggable）
       → 不可拖拽则直接 return（库基于旧 list 继续，不报错）
    ② 重排 displayItems（removeAt + add）
    ③ 实时检测置顶区跨越 → 更新 crossedPinnedZone
    ④ 触觉反馈（节流）
})
└─ LazyColumn(state = lazyListState) {
      itemsIndexed(displayItems, key) { index, item ->
         ReorderableItem(
            state = reorderableState,
            key = key(item),
            enabled = isDragEnabled && isDraggable(item)   // 控制可重排序性
         ) { isDragging ->
            val handleModifier = Modifier.longPressDraggableHandle(
                enabled = isDragEnabled && isDraggable(item),  // 控制可拖动性
                onDragStarted = { 触觉反馈; isDragActive = true; 记录 draggedOriginalIndex },
                onDragStopped = { 提交 onReorder; 触觉反馈; isDragActive = false }
            )
            if (isDragging) {
                // 虚线占位框（Canvas）+ 浮起卡片（alpha 0.7 + scale 1.05 + shadow 8dp）
                // handleModifier 应用在浮起卡片上
            } else {
                // 普通项，animateItem() 让位动画（库默认 spring）
            }
         }
      }
   }
```

### ReorderableColumn 适配流

```
项目的 ReorderableColumn(items, onReorder, modifier, content)
└─ 库的 ReorderableColumn(
      list = items,
      onSettle = onReorder,
      modifier = modifier,
      content = { index, item, isDragging ->
         ReorderableColumnScope.ReorderableItem {
            // 包装项目 content 调用
            content(index, item, isDragging)
         }
      }
   )
```

### 提交时机

- 库的 `onMove` 在拖拽中每次交换时触发，立即更新 `displayItems`
- `onDragStarted` 记录 `draggedOriginalIndex`（被拖项在原始 items 中的索引）
- `onMove` 中实时计算 `crossedPinnedZone`（调用 `ReorderAlgorithms.checkPinnedZoneCrossed`）
- 松手时 `onDragStopped` 回调中，比较 `draggedOriginalIndex` 与 `displayItems` 中被拖项的最终索引，调用 `onReorder(fromIndex, toIndex, crossedPinnedZone)` 一次性提交

## 六、关键问题解决机制

| 问题 | 旧根因 | 新机制（库内置） | 源码位置 |
|---|---|---|---|
| 问题1（首项下拖失控） | `fingerY` 在顶部边缘误触发向上 auto-scroll | 基于 `handleOffset`（手柄中心）距视口边缘触发，首项下拖时 handleOffset 增大远离顶部 | `ReorderableLazyCollection.kt:443-482` |
| 问题2（交换首项飞出） | `draggedListCenterY` 用逻辑索引×高度，与视口 `fingerY` 不同坐标系 | `draggingItemOffset = draggedDelta + (initialOffset - currentOffset)`，全部视口坐标 | `ReorderableLazyCollection.kt:316-331` |
| 首可见项跳变 | 自研无法支持 | `moveItems` 中 `requestScrollToItem` 保持首项位置 | `ReorderableLazyCollection.kt:632-639` |
| 释放回弹动画 | 自研受限挂起作用域无法调用 | `previousDraggingItemOffset.animateTo` 内置 spring | `ReorderableLazyCollection.kt:396-407` |
| 移动后一帧跳变 | 自研读陈旧 visibleItemsInfo | `predictedDraggingItemOffset` 预测新位置 | `ReorderableLazyCollection.kt:645-649` |

## 八、风险点与应对

| 风险 | 影响 | 应对 | 源码依据 |
|---|---|---|---|
| CompletedDivider 不可拖拽项 | 库 onMove 的 index 含所有项，divider 可能被交换 | `onMove` 中校验 `isDraggable(displayItems[from.index]) && isDraggable(displayItems[to.index])`，不可拖拽则直接 return | `onMove` 是 suspend，return 后库基于旧 list 继续 |

## 十三、多选合并拖拽设计

### 13.1 背景

Calvin-LL/Reorderable 库仅支持**单项**拖拽排序（`longPressDraggableHandle` 绑定到单个 item），不原生支持"多选合并拖拽"。多选合并拖拽指：用户在多选模式下选中多项后，长按其中一项触发拖拽，所有选中项作为一个整体移动，释放时按拖拽位置批量重排。

正式实现需在库之上**自定义包装**，核心思路：**隐藏已选项 + 浮层单卡 pointerInput + 释放后批量 reorder 列表**。参考 HTML 原型 `docs/superpowers/prototypes/拖拽排序重构原型.html` 中已验证的实现。

### 13.2 交互流程

| 阶段 | 触发条件 | 行为 |
|---|---|---|
| 进入多选 | 非多选模式长按 500ms + 抬手（未移动超过 touchSlop） | `enterBatchMode(item)`：isBatchMode=true，selectedIds={item.id} |
| 选择多项 | 多选模式下点击卡片或勾选框 | `toggleSelection(itemId)`：增删 selectedIds 集合 |
| 触发合并拖拽 | 多选模式下长按**已选中**项 500ms | `selectedIds.size > 1` → `startMergeDrag(anchor)`；`size == 1` 回退普通单项拖拽 |
| 拖拽中 | 手指移动 | 合并卡片跟随手指，占位框（单项高度）在列表中定位 |
| 释放 | 手指抬起 | `endDrag` 合并分支：cross-fade + transform 补偿同步释放 |
| 清理 | 动画结束 | `cleanupMergeDrag`：刷新状态，保留多选模式 |
| 退出多选 | 点击顶部栏关闭按钮 | `exitBatchMode()`：清空 selectedIds |

### 13.3 自定义包装方案

库不原生支持合并拖拽，在 `ReorderableLazyColumn` 之上包装一层"合并拖拽模式"。

#### 状态扩展

在 ReorderableLazyColumn 内部状态基础上新增：

```kotlin
var isBatchMode by remember { mutableStateOf(false) }
var selectedIds by remember { mutableStateOf(setOf<Any>()) }
var isMergeDragging by remember { mutableStateOf(false) }
var mergeSelectedItems by remember { mutableStateOf(listOf<T>()) }
```

#### 手势分层

```
非多选模式：
  库 longPressDraggableHandle → 长按 500ms 直接开始拖拽
  （长按 500ms + 抬手未移动 → 进入多选模式）

多选模式：
  自定义 pointerInput（捕获阶段拦截）：
    - 长按已选中项 500ms
    - selectedIds.size > 1 → 进入合并拖拽（禁用库的 longPressDraggableHandle）
    - selectedIds.size == 1 → 回退库的普通单项拖拽
    - 长按未选中项 → 不触发拖拽（交给 click 处理 toggleSelection）
```

合并拖拽期间 `isDragEnabled = false`，禁用库的 `longPressDraggableHandle`，改为自定义 `pointerInput { detectDragGesturesAfterLongPress }` 驱动合并卡片。

#### 合并拖拽实现（4 步）

**Step 1：startMergeDrag(anchor)**
- 收集所有选中项 `mergeSelectedItems`（按 LazyList 可见项顺序）
- 在 anchor 原位置插入占位框（**单项高度**，非组合总高度）
- 折叠隐藏所有选中项（height=0, alpha=0, 等效 `merge-hidden`）
- 创建浮层"选中 N 项"合并卡片，`Modifier.offset` 跟随手指
- 合并卡片脱离 LazyList，加入外层 Box 浮层

**Step 2：拖拽中**
- 合并卡片跟随手指（自定义 pointerInput，非库的拖拽）
- 占位框根据手指位置在列表中定位（复用 `updatePlaceholderPosition` 算法）
- 边缘自动滚动（复用库的 scroller）
- **占位框边界限制**：任何待办的占位框都不能插入到"置顶"标签上方

**Step 3：endDrag（方案 C：transform 补偿同步释放）**

释放时原项展开会引发布局跳动，采用 FLIP 变体消除：

1. 合并卡片弹簧动画到占位框位置 + 淡出（0.35s）
2. 立即将原项插入到占位框位置 + 取消隐藏（`alpha` 锁定为 0）
3. 移除占位框
4. 给后续项加 `graphicsLayer { translationY = -(N-1) × itemHeight }` 抵消原项展开位移
5. 下一帧清除 `alpha` 锁定 + `translationY` → 触发原项淡入（0→1, 0.2s）+ 后续项下移过渡（0.3s）
6. cross-fade：合并卡片淡出 ↔ 原项淡入同步启动

**Step 4：cleanupMergeDrag**
- 移除合并卡片
- 按新位置刷新完成/置顶状态与分区计数
- 重置合并拖拽状态（**保留多选模式**，仍处于批量态）

### 13.4 关键技术点

| 问题 | 方案 | 参考原型 |
|---|---|---|
| 库不支持合并拖拽 | 自定义 pointerInput 包装，合并拖拽时 `isDragEnabled=false` 禁用库 | `startMergeDrag` |
| 占位框高度 | 单项高度（非组合总高度），释放时用 transform 补偿消除跳动 | 方案 C |
| `!important` 与 transition 冲突 | Compose 用 `AnimatedVisibility` / `Animatable` 替代，无此问题 | 原型用 inline style 锁定 + 双 rAF |
| cross-fade 同步 | 合并卡片 0.35s 淡出 + 原项 0.2s 淡入同步启动 | `endDrag` 合并分支 |
| 单选回退 | `selectedIds.size==1` 时调用普通单项拖拽，避免"选中 1 项"突兀 | `pointerdown` 多选分支 |
| 置顶区边界 | 占位框不能插入到"置顶" section-label 之前 | `updatePlaceholderPosition` |
| 抑制回旋 click | pointerup 进入多选后，浏览器派发 click 需拦截 | `suppressNextClick` 捕获阶段拦截 |

### 13.5 占位框边界限制

`updatePlaceholderPosition` 算法中，任何待办（无论置顶与否）的占位框都不能插入到"置顶"标签上方：

```
1. 收集所有可比较子元素（todo-item、section-label、completed-divider）
2. 找到第一个中心 Y > 被拖项中心 Y 的元素作为插入点
3. 边界限制：若插入点在"置顶"标签之前或就是它，强制限制到"置顶"标签之后
4. 非置顶项仍可拖入置顶区（拖入后由 updateItemPinnedState 自动切换为置顶项）
```

### 13.6 Compose 实现与 HTML 原型对应关系

| HTML 原型 | Compose 实现 |
|---|---|
| `merge-hidden` CSS 类（max-height:0, opacity:0） | `Modifier.height(0.dp).alpha(0f)` 或 `AnimatedVisibility` |
| 合并卡片 `position:fixed` 跟随手指 | `Box(Modifier.offset { IntOffset(0, fingerY) })` 浮层 |
| `updatePlaceholderPosition` 比较中心 Y | 自研位置计算或库的 `onMove` |
| transform 补偿（`translateY(-offset)`） | `Modifier.graphicsLayer { translationY = -offset }` |
| 双 rAF 清除 inline opacity | `LaunchedEffect` + `Animatable.animateTo` |
| `cross-fade` | `AnimatedVisibility` 或 `Crossfade` |
| `suppressNextClick` 捕获阶段拦截 | Compose 无 click 冒泡问题，pointerInput 优先级控制 |
