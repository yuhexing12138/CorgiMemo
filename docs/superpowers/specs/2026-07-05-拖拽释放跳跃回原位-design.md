## 1. 背景与问题

### 1.1 用户报告的现象

**任意一个待办卡片**拖拽释放后，都会**跳跃回原位**。同时，另一项会被错误地"消失"或"移到错误位置"。

### 1.2 影响范围

- **影响版本**：Zone 状态机拖拽架构重构后（commit `dd816f8` 起）
- **影响场景**：所有拖拽场景（同区拖拽、跨区拖拽）
- **影响程度**：拖拽功能完全不可用，视觉与持久化均错误

## 2. 根因分析

### 2.1 触发链路

```
1. 原始列表：[A, B, C]   (HomeScreen.displayItems 与组件内部 displayItems 一致)
2. 用户长按 A，拖到 B 和 C 之间
3. onMove 触发：组件内部 displayItems = [B, A, C]
   ↳ HomeScreen.displayItems 仍是 [A, B, C]（remember 依赖未变）
4. 释放手指 → onDragStopped
5. Calvin-LL/Reorderable 库的 onDragStop() 启动 settle 动画
6. 我们的 onDragStopped lambda 执行：
   - draggedCurrentIndex = 1（组件内部 displayItems 中 A 的位置）
   - onReorder(dragResult, fromIndex=0, toIndex=1)
7. HomeScreen.onReorder 执行：
   - displayItems.getOrNull(1) = B  ← ❌ 取错项！应该是 A
   - draggedTodo = B
   - relativeIndex 基于 [A, B, C] 计算（错）
8. viewModel.reorderOnDragResult(draggedItemId=B.id, ...)
   - 把 B 移到 target zone 的 relativeIndex
   - A 留在原位（PENDING 区顶部）
9. 持久化完成 → StateFlow 推送新数据
10. HomeScreen.displayItems 重新计算 → 传入 ZonedReorderableLazyColumn
11. LaunchedEffect(items) 触发：displayItems = items
12. LazyColumn 基于 key 重新对齐 → A 瞬移回原位
    ↳ settle 动画被打断，视觉表现为"跳跃回原位"
```

### 2.2 根本原因

`HomeScreen.onReorder` 中的 `displayItems` 是 **HomeScreen 闭包变量**，由 `remember(filteredPending, filteredCompleted, ...)` 计算，**在拖拽过程中不会更新**。

而 `ZonedReorderableLazyColumn` 内部维护了另一个 `displayItems`（`var displayItems by remember { mutableStateOf(items) }`），它在 `onMove` 中被实时更新（`removeAt + add`）。

**两个 displayItems 不同步**导致 HomeScreen 取错被拖项 + 计算 relativeIndex 错误。

## 3. 设计方案

### 3.1 方案选型

| 方案 | 描述 | 优点 | 缺点 |
|------|------|------|------|
| **方案 A（采用）** | 扩展 onReorder 签名，由组件计算并传递被拖项 + relativeIndex | 职责清晰，类型安全，最小暴露 | onReorder 签名变更（破坏性） |
| 方案 B | 传递 finalDisplayItems 给 HomeScreen | 保留 HomeScreen 控制权 | 暴露组件内部状态，HomeScreen 仍需重复计算 |
| 方案 C | 用 key 识别被拖项，HomeScreen 反向查找 | — | relativeIndex 仍需 finalDisplayItems，退化为方案 B |

### 3.2 选型理由

1. **职责匹配**：组件已经在 onMove 中维护 displayItems，最了解项的位置
2. **类型安全**：`DisplayItem.Todo` 类型化，避免 Any key 反查
3. **最小暴露**：不暴露组件内部 displayItems 状态
4. **符合现有架构**：Zone 状态机已封装了 zone 推导，索引计算也应由组件负责
5. **HomeScreen 简化**：从"取项 + 计算 relativeIndex + 调用 ViewModel"简化为"纯转发"

### 3.3 架构变更

#### 修改前

```
┌────────────────────────────┐         ┌──────────────────┐
│ ZonedReorderableLazyColumn │         │   HomeScreen     │
│                            │         │                  │
│  onMove → 更新内部状态       │         │  onReorder:      │
│  onDragStopped:             │ ──────> │   取项（错！）    │
│    onReorder(result,        │         │   计算 relative  │
│       fromIdx, toIdx)       │         │      （错！）    │
│                            │         │   调用 ViewModel  │
└────────────────────────────┘         └──────────────────┘
```

#### 修改后

```
┌────────────────────────────┐         ┌──────────────────┐
│ ZonedReorderableLazyColumn │         │   HomeScreen     │
│                            │         │                  │
│  onMove → 更新内部状态       │         │  onReorder:      │
│  onDragStopped:             │         │   纯转发给       │
│    计算 draggedItem         │ ──────> │   ViewModel      │
│    计算 relativeIndex       │         │                  │
│    onReorder(result,        │         │                  │
│       draggedItem,          │         │                  │
│       relativeIndex)        │         │                  │
└────────────────────────────┘         └──────────────────┘
```

**关键变化**：
1. **职责上移**：被拖项查找 + relativeIndex 计算 从 HomeScreen 移到 ZonedReorderableLazyColumn
2. **数据源正确**：基于组件内部 displayItems（已被 onMove 更新）计算
3. **HomeScreen 简化**：从"取项 + 计算 + 调用 ViewModel"简化为"纯转发"

## 4. 组件与接口

### 4.1 onReorder 签名变更

```kotlin
// 修改前
onReorder: (dragResult: ZoneDragResult, fromIndex: Int, toIndex: Int) -> Unit

// 修改后
onReorder: (
    dragResult: ZoneDragResult,
    draggedItem: DisplayItem.Todo,
    targetZoneRelativeIndex: Int
) -> Unit
```

**变更点**：
- `fromIndex: Int` ❌ 删除
- `toIndex: Int` ❌ 删除
- `draggedItem: DisplayItem.Todo` ✅ 新增（被拖项，类型化）
- `targetZoneRelativeIndex: Int` ✅ 新增（目标 zone 内相对索引）

### 4.2 为什么用 `DisplayItem.Todo` 而不是 `TodoItem`

| 选项 | 优点 | 缺点 |
|------|------|------|
| `DisplayItem.Todo`（采用） | 类型安全，与组件内部数据结构一致 | HomeScreen.onReorder 需通过 `.item` 取 `TodoItem` |
| `TodoItem` | HomeScreen 直接拿到 TodoItem | 调用方可能误传非 Todo 项的 TodoItem；丢失 DisplayItem 上下文 |

### 4.3 relativeIndex 计算规则

```kotlin
val targetZone = dragResult.currentZone
var relativeIndex = 0
for (i in 0 until draggedCurrentIndex) {
    val item = internalDisplayItems[i]
    if (item is DisplayItem.Todo && item.item.zone() == targetZone) {
        relativeIndex++
    }
}
```

**关键点**：
1. 基于 **内部 displayItems**（已被 onMove 更新）计算
2. `draggedCurrentIndex` 是被拖项在内部 displayItems 中的当前位置
3. 仅统计**同 zone 的 Todo 项**，跳过 divider 与其他 zone 的 Todo
4. 与 HomeScreen 原有算法一致，仅数据源切换

### 4.4 内部辅助函数

将 relativeIndex 计算逻辑提取为**内部可见的纯函数**，便于测试：

```kotlin
internal fun computeRelativeIndexInZone(
    displayItems: List<DisplayItem>,
    draggedCurrentIndex: Int,
    targetZone: TodoZone
): Int {
    var relativeIndex = 0
    for (i in 0 until draggedCurrentIndex) {
        val item = displayItems[i]
        if (item is DisplayItem.Todo && item.item.zone() == targetZone) {
            relativeIndex++
        }
    }
    return relativeIndex
}
```

## 5. 组件实现变更

### 5.1 onDragStopped 修改前

```kotlin
onDragStopped = {
    val draggedCurrentIndex = displayItems.indexOfFirst {
        key(it) == key(item)
    }
    if (draggedOriginalIndex >= 0 &&
        draggedOriginalIndex != draggedCurrentIndex &&
        draggedCurrentIndex >= 0
    ) {
        val dragResult = dragZoneState.endDrag()
        onReorder(
            dragResult,
            draggedOriginalIndex,    // ❌ HomeScreen 用不到
            draggedCurrentIndex       // ❌ 用于取项（取错）
        )
        HapticFeedbackManager.performHapticFeedback(...)
    }
    isDragActive = false
    draggedOriginalIndex = -1
    dragZoneState.reset()
}
```

### 5.2 onDragStopped 修改后

```kotlin
onDragStopped = {
    val draggedCurrentIndex = displayItems.indexOfFirst {
        key(it) == key(item)
    }
    if (draggedOriginalIndex >= 0 &&
        draggedOriginalIndex != draggedCurrentIndex &&
        draggedCurrentIndex >= 0
    ) {
        val dragResult = dragZoneState.endDrag()

        // ✅ 基于内部 displayItems（已被 onMove 更新）取被拖项
        val draggedTodoItem = displayItems[draggedCurrentIndex] as? DisplayItem.Todo
            ?: return@longPressDraggableHandle

        // ✅ 基于内部 displayItems 计算 target zone 相对索引
        val targetZone = dragResult.currentZone
        val relativeIndex = computeRelativeIndexInZone(
            displayItems, draggedCurrentIndex, targetZone
        )

        onReorder(dragResult, draggedTodoItem, relativeIndex)
        HapticFeedbackManager.performHapticFeedback(...)
    }
    isDragActive = false
    draggedOriginalIndex = -1
    dragZoneState.reset()
}
```

### 5.3 onDragStarted 同步优化

`onDragStarted` 中已经从内部 displayItems 取 `draggedTodoItem`（[ZonedReorderableLazyColumn.kt:147-148](file:///c:/Users/EDY/Desktop/CorgiMemo/app/src/main/java/com/corgimemo/app/ui/components/ZonedReorderableLazyColumn.kt#L147-148)），无需修改，行为正确。

### 5.4 关键不变式

1. **被拖项的 key 不变**：`draggedCurrentIndex` 通过 `key(it) == key(item)` 查找
2. **relativeIndex 算法一致**：与 HomeScreen 原有逻辑相同，仅数据源切换
3. **divider 跳过**：`item_i is DisplayItem.Todo` 自动跳过 divider
4. **跨 zone 计数**：`item_i.item.zone() == targetZone` 跳过其他 zone 的 Todo

### 5.5 边界场景验证

| 场景 | 行为 |
|------|------|
| 拖到列表开头（divider 前） | `draggedCurrentIndex = 0`，循环不执行，`relativeIndex = 0` ✅ |
| 拖到目标 zone 第一项前 | `relativeIndex = 0`，被拖项成为该 zone 第一项 ✅ |
| 同 zone 内重排 | `relativeIndex` 正确反映 zone 内位置 ✅ |
| 跨 zone 拖拽 | `targetZone = currentZone`，仅统计目标 zone 的 Todo ✅ |

## 6. 调用方变更

### 6.1 HomeScreen.onReorder 修改前

```kotlin
onReorder = { dragResult, fromIndex, toIndex ->
    // ❌ 这里的 displayItems 是 HomeScreen 闭包变量，未在拖拽中更新
    val draggedTodoItem = displayItems.getOrNull(toIndex) as? DisplayItem.Todo
    val draggedTodo = draggedTodoItem?.item
        ?: return@ZonedReorderableLazyColumn

    // ❌ 同样基于错误的 displayItems
    val targetZone = dragResult.currentZone
    var relativeIndex = 0
    for (i in 0 until toIndex) {
        val item = displayItems[i]
        if (item is DisplayItem.Todo && item.item.zone() == targetZone) {
            relativeIndex++
        }
    }

    viewModel.reorderOnDragResult(
        draggedItemId = draggedTodo.id,
        draggedTodo = draggedTodo,
        dragResult = dragResult,
        targetZoneRelativeIndex = relativeIndex
    )
}
```

### 6.2 HomeScreen.onReorder 修改后

```kotlin
onReorder = { dragResult, draggedTodoItem, targetZoneRelativeIndex ->
    // ✅ 纯转发：组件已基于内部 displayItems 计算好被拖项和相对索引
    viewModel.reorderOnDragResult(
        draggedItemId = draggedTodoItem.item.id,
        draggedTodo = draggedTodoItem.item,
        dragResult = dragResult,
        targetZoneRelativeIndex = targetZoneRelativeIndex
    )
}
```

## 8. 风险与缓解

### 8.1 风险 1：DisplayItem.Todo 类型转换失败

**场景**：`displayItems[draggedCurrentIndex]` 在极端情况下不是 `DisplayItem.Todo`（例如 divider 被错误地拖动）。

**概率**：低。`onMove` 中已有 `isDraggable` 校验，divider 不可拖拽、不可作为目标。

**缓解**：使用安全转换 `as?` + early return，转换失败时静默退出，不抛异常。

### 8.2 风险 2：relativeIndex 边界越界

**场景**：`draggedCurrentIndex = 0` 时循环不执行，`relativeIndex = 0`，正确。但若 `draggedCurrentIndex` 异常大可能产生错误索引。

**缓解**：`HomeViewModel.reorderOnDragResult` 已有 `coerceIn(0, targetList.size)` 防御（[HomeViewModel.kt:866](file:///c:/Users/EDY/Desktop/CorgiMemo/app/src/main/java/com/corgimemo/app/viewmodel/HomeViewModel.kt#L866)）。

### 8.3 风险 3：onReorder 签名破坏性变更

**场景**：`onReorder` 签名变更可能影响其他调用方。

**排查**：已确认只有 [HomeScreen.kt:811](file:///c:/Users/EDY/Desktop/CorgiMemo/app/src/main/java/com/corgimemo/app/ui/screens/home/HomeScreen.kt#L811) 一个调用方。

**缓解**：编译期即可发现所有未更新的调用方。

### 8.4 风险 4：settle 动画与数据更新竞态

**场景**：Calvin-LL/Reorderable 库的 `onDragStop()` 会启动 settle 动画，同时 HomeViewModel 异步持久化后 StateFlow 推送新数据，可能打断动画。

**现状**：本次修改**不引入新风险**。该竞态在修改前已存在，且 `LaunchedEffect(items) → displayItems = items` 的同步策略已能正确处理。

**缓解**：本次修改通过消除"取错项"的根因，反而**降低**了 settle 动画被打断的概率。

### 8.5 风险评估总结

| 风险 | 概率 | 影响 | 缓解策略 |
|------|------|------|----------|
| 1. 类型转换失败 | 极低 | 静默退出 | `as?` + early return |
| 2. relativeIndex 越界 | 极低 | 落到错误位置 | ViewModel 已有 coerceIn 防御 |
| 3. 签名破坏性 | 已确认无 | N/A | 编译期发现 |
| 4. settle 动画竞态 | 已存在 | 视觉跳跃 | 本次修改降低风险 |
