# 待办卡片多容器分类独立设置修复设计

> 创建日期：2026-06-28
> 状态：待实施
> 关联模块：待办编辑页（TodoEditScreen / TodoEditViewModel / CheckboxEditText）

## 一、背景与问题

### 1.1 问题描述

在 `c:\Users\Lenovo\Desktop\CorgiMemo` 项目的待办卡片编辑页面中，当用户在输入框中输入 "/" 后会创建多个独立的待办容器（按 `TodoLine.groupId` 分组）。此时存在分类设置异常：

> 当用户在第一个容器中设置分类为"工作"后，后续所有容器的分类会被统一更改为"工作"。

### 1.2 根因分析

`TodoEditViewModel` 中的分类状态使用单个 `MutableStateFlow<Long>` 存储，被所有分组容器共享：

```kotlin
private val _categoryId = MutableStateFlow(0L)
val categoryId: StateFlow<Long> = _categoryId.asStateFlow()
```

而同模块的其他分组级状态（提醒、优先级、重复类型、保存状态）都已正确使用 `Map<Int, ...>` 按 `groupId` 隔离。**分类状态是唯一例外**，导致一处修改影响所有容器。

### 1.3 影响范围

| 影响点 | 表现 |
|--------|------|
| UI 渲染 | 所有容器的"分类"按钮同步显示同一分类名 |
| 用户操作 | 在任一容器选择/清除分类，所有容器同步变化 |
| 数据持久化 | 多容器保存时，所有 todo 写入同一 categoryId |

### 1.4 用户要求

1. 每个容器的分类选择状态应独立存储
2. 选择某一容器的分类时，其他容器的分类状态保持不变
3. 分类设置后应即时保存并正确显示在对应容器中

## 二、方案对比与选择

### 2.1 三种候选方案

| 维度 | 方案A：完全按分组隔离 | 方案B：保留旧字段+分组覆盖层 | 方案C：UI层 rememberSaveable |
|------|---------------------|---------------------------|---------------------------|
| 数据结构 | `_groupCategoryIds: Map<Int, Long>` | 保留 `_categoryId` + 新增 `_groupCategoryOverrides: Map<Int, Long>` | ViewModel 不动，UI 用 `rememberSaveable` |
| 与现有模式一致性 | 完全一致（_groupReminders/_groupPriorities 同模式） | 双数据源，与现有模式不一致 | 完全偏离，状态分散 |
| 改动范围 | ViewModel + TodoEditScreen + CheckboxEditText 三层 | ViewModel 双源逻辑 + UI 部分改动 | 仅 UI 层改动 |
| 歧义风险 | 无 | "未设置"vs"显式设为0L"难区分 | rememberSaveable 对 Map 序列化不稳定，进程重建风险 |
| 长期维护成本 | 低 | 中（双源需同步） | 高（状态分散，违反单一数据源） |
| YAGNI | 符合 | 过度设计兼容层 | 符合但架构偏离 |

### 2.2 选定方案

**采用方案A：完全按分组隔离**。

**理由**：
1. 与已有的 `_groupReminders`/`_groupPriorities`/`_groupRepeatTypes`/`_groupSaveStates` 完全同模式，架构统一，零认知成本
2. 单一数据源，每个 groupId 对应唯一 categoryId，零歧义
3. 旧的 `_categoryId`/`setCategoryId`/`clearCategory` 仅被 `TodoEditScreen` 调用（经 grep 确认），可彻底删除；`InspirationEditViewModel` 有自己独立副本，互不影响
4. 完全满足用户三项要求

### 2.3 关键设计决策

| 决策点 | 选择 | 理由 |
|--------|------|------|
| 新容器默认分类 | 0L（未分类） | 用户确认；与 `_groupReminders` 等分组状态初值（0/null）一致；最纯粹地体现"独立设置"语义 |
| 0L 的语义 | 未分类 | 与 `TodoItem.categoryId` 默认值一致；与旧 `clearCategory` 实现一致；无需引入 nullable |
| 旧 `_categoryId` 字段 | 删除 | 仅 `TodoEditScreen` 调用，无外部依赖 |
| 弹窗状态 | `pendingCategoryGroupId: Int?` 替代 `showCategoryDialog: Boolean` | null=弹窗关闭，非 null 携带目标 groupId |
| `TodoGroupContainer` 签名 | 不变 | 已接收 `groupId` 参数，"按 groupId 取值"职责放在外层，保持"渲染单个容器"单一职责 |

## 三、整体架构与数据流

### 3.1 数据流总览

```
┌──────────────────────────────────────────────────────────────┐
│  TodoEditViewModel (单一数据源)                                │
│  ─────────────────────────────────────────                   │
│  _groupCategoryIds: MutableStateFlow<Map<Int, Long>>          │
│      key = groupId, value = categoryId (0L = 未分类)          │
│                                                               │
│  方法:                                                        │
│   - setGroupCategory(groupId, categoryId)                     │
│   - clearGroupCategory(groupId)         // 置为 0L            │
│  ─────────────────────────────────────────                   │
│  _groupReminders / _groupPriorities / _groupRepeatTypes       │
│  （已有，模式完全一致，作为参照）                                │
└──────────────────────────────────────────────────────────────┘
            ▲                              │
            │ 用户选择分类                  │ collectAsState
            │                              ▼
┌──────────────────────────────────────────────────────────────┐
│  TodoEditScreen                                              │
│  ─────────────────────────────────────────                   │
│  pendingCategoryGroupId: Int?   // null=弹窗关闭              │
│  groupCategoryIds by viewModel.groupCategoryIds.collectAsState│
│  groupCategoryNames: 派生 (按 groupId 查 categories)          │
│                                                               │
│  onCategoryClick = { groupId -> pendingCategoryGroupId = groupId }│
│  onCategoryClear = { groupId -> viewModel.clearGroupCategory(groupId) }│
└──────────────────────────────────────────────────────────────┘
            ▲                              │
            │ 透传 map + 携带 groupId 的回调  │
            │                              ▼
┌──────────────────────────────────────────────────────────────┐
│  CheckboxEditText                                            │
│  ─────────────────────────────────────────                   │
│  按 groupId 分组渲染 TodoGroupContainer:                      │
│    categoryId = groupCategoryIds[groupId] ?: 0L              │
│    categoryName = groupCategoryNames[groupId]                │
│    onCategoryClick = { onCategoryClick?.invoke(groupId) }    │
└──────────────────────────────────────────────────────────────┘
```

### 3.2 核心数据流路径

| 场景 | 数据流 |
|------|--------|
| 用户在容器B点击分类按钮 | `TodoGroupContainer.onCategoryClick` → `CheckboxEditText` 透传 → `TodoEditScreen` 设置 `pendingCategoryGroupId = B` → 弹窗打开 |
| 用户在弹窗选择"工作" | `CategorySelectorDialog.onCategorySelected(工作ID, "工作")` → `viewModel.setGroupCategory(B, 工作ID)` → `_groupCategoryIds[B] = 工作ID` → UI 重组，仅容器B显示"工作" |
| 用户在容器B点击×清除 | `TodoGroupContainer.onCategoryClear` → `viewModel.clearGroupCategory(B)` → `_groupCategoryIds[B] = 0L` → 仅容器B回到"分类"占位 |
| 保存 | `buildTodoItemForGroup(targetGroupId)` 读取 `_groupCategoryIds.value[targetGroupId] ?: 0L` 写入 `TodoItem.categoryId` |
| 加载已有 todo | `loadTodo` 中 `_groupCategoryIds.value = mapOf(0 to todo.categoryId)`（单容器场景 groupId=0） |
| 输入"/"新建容器 | `onNewGroupRequested` 创建新 `TodoLine(groupId = maxGroupId+1)`，新 groupId 不在 map 中 → 渲染时 `groupCategoryIds[newGroupId] ?: 0L` 默认未分类 |

### 3.3 三项用户要求映射

| 用户要求 | 设计保证 |
|---------|---------|
| 1) 每个容器的分类选择状态应独立存储 | `_groupCategoryIds: Map<Int, Long>` 按 groupId 独立键值存储 |
| 2) 选择某一容器的分类时，其他容器的分类状态保持不变 | `setGroupCategory(groupId, id)` 只修改 map 中该 groupId 的条目，其他条目不动 |
| 3) 分类设置后应即时保存并正确显示在对应容器中 | StateFlow + Compose `collectAsState` 自动重组；`TodoGroupContainer` 按 groupId 取对应 categoryId/categoryName 渲染 |

## 五、数据持久化与兼容性

### 5.1 持久化路径

| 场景 | 数据流 | 数据库影响 |
|------|--------|----------|
| 单容器保存（existingTodo） | `performSave` → `categoryId = _groupCategoryIds.value[0] ?: 0L` → 写入 `TodoItem.categoryId` | 与旧逻辑等价 |

| 加载已有 todo | `loadTodo` → `_groupCategoryIds.value = mapOf(0 to todo.categoryId)` | 无写入，仅状态初始化 |

### 5.2 数据模型 — 零改动

- `TodoItem` 实体：`categoryId: Long` 字段不变
- `Category` 实体：不变
- 数据库 schema：不变
- Migration：**不需要**（仅 UI/ViewModel 层改造）

### 5.3 兼容性

- **旧 todo 数据**：每个 todo 已有自己的 categoryId，加载时正确写入 `_groupCategoryIds[0]`，100% 兼容
- **CSV/ICS 导出**：使用 `todo.categoryId` 实体字段，不受影响
- **Widget/通知**：使用 `todo.categoryId` 实体字段，不受影响
- **HomeScreen 显示**：使用 `todo.categoryId` 实体字段，不受影响

## 六、边界情况与错误处理

### 6.1 边界情况

| 边界情况 | 处理策略 |
|---------|---------|
| 新容器未设置分类 | `groupCategoryIds[groupId]` 返回 null → `?: 0L` fallback，渲染"分类"占位，保存写入 0L |
| 用户清除某容器分类 | `clearGroupCategory(groupId)` 写入 `groupId to 0L`，与"从未设置"在 UI 上等价（都显示"分类"占位） |
| 用户删除某容器 | 该 groupId 不再渲染，map 中残留条目无副作用；下次新建容器复用 `maxGroupId+1` 不会冲突 |
| 快速连续输入 "/" 创建多容器 | 每个 `onNewGroupRequested` 生成新 groupId，map 不预置条目，渲染时 fallback 到 0L，无竞态 |
| `loadTodo` 后用户输入 "/" 新建第二容器 | 新 groupId 不在 map 中（map 仅含 `0 to todo.categoryId`），新容器默认 0L 未分类 ✓ |
| 进程重建（旋转屏幕等配置变更） | Compose `remember` 状态保留；ViewModel 由 ViewModelProvider 保持，`_groupCategoryIds` 不丢失 |
| 进程被杀后恢复 | ViewModel 状态丢失，但已保存的 todo 数据已持久化；未保存的编辑态分类丢失（与现有 `_groupReminders` 行为一致，可接受） |
| 自定义分类创建失败 | `createCustomCategory` 回调不触发，`setGroupCategory` 不被调用，状态保持原值 |
| categoryId 引用了已删除的 Category | `categories.find { it.id == catId }?.name` 返回 null，渲染"分类"占位；保存时仍写入原 catId（独立问题，不在本次修复范围） |

### 6.2 错误处理

本次改造为状态管理重构，**无新增 IO 操作**，错误面极小：

| 潜在错误点 | 处理 |
|-----------|------|
| `groupCategoryIds[groupId]` 返回 null | `?: 0L` 兜底，永远有值 |
| `categories.find { it.id == catId }?.name` 返回 null | `categoryName ?: "分类"` 占位兜底（已有逻辑） |
| `createCustomCategory` 异常 | 由 ViewModel 内部 try-catch 处理（已有逻辑），不影响 `_groupCategoryIds` |
| 用户在弹窗打开期间删除容器 | `pendingCategoryGroupId` 仍指向已删除 groupId，`setGroupCategory` 写入 map 后无 UI 消费者，无副作用 |
