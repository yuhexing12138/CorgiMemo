# 待办卡片长按交互优化设计

> 创建日期：2026-06-27
> 状态：已确认，待实施

## 一、背景与目标

### 1.1 当前行为

当前待办卡片长按交互使用 `detectTapGestures` 的 `onLongPress` 回调：

- 长按达到 500ms 阈值时**立即触发**（手指仍在屏幕上）
- 调用 `onLongClick()` → `viewModel.enterBatchMode(todo.id)`
- `enterBatchMode` 已同时完成：进入批量模式 + 选中该待办

### 1.2 问题

- 长按立即触发，与用户预期（抬起手指后确认）不符
- 长按触发后手指移动无法取消，可能与左滑手势冲突

### 1.3 目标

- 长按达到阈值后，**手指完全离开屏幕**才执行勾选+多选
- 长按达到阈值后，手指移动超过阈值则取消（让左滑接管）
- 长按阈值 ≥ 500ms，避免误触
- 勾选状态更新与多选页面显示同步完成

## 二、方案选择

### 2.1 对比方案

| 方案 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| 方案1：`awaitEachGesture` 统一手势检测（选用） | 替换 `detectTapGestures`，完全自定义手势状态机 | 满足全部需求，与项目已有模式一致 | 改动较大，需手动处理水波纹 |
| 方案2：`detectTapGestures` + `tryAwaitRelease` 协调 | `onLongPress` 设标志，`onPress` 抬起后执行 | 改动小 | 不支持"长按后移动取消"，与左滑冲突 |
| 方案3：容器层 `awaitEachGesture` | 在 SwipeableTodoBox 层统一处理 | 手势冲突处理最干净 | 改动范围大，SwipeableTodoBox 职责膨胀 |

### 2.2 选定方案

**方案1**。理由：

1. 满足全部需求（抬起触发 + 移动取消）
2. 与项目已有的 `awaitEachGesture` 模式一致（`ReorderableLazyColumn.kt` 已用此模式）
3. 改动集中在 `TodoListItem.kt`，不影响 `SwipeableTodoBox`

## 三、改动范围

| 文件 | 改动内容 |
|------|---------|
| `app/src/main/java/com/corgimemo/app/ui/components/TodoListItem.kt` | 替换 Card 的 `pointerInput` 中 `detectTapGestures` 为 `awaitEachGesture` |

**不改动**：

- `HomeScreen.kt`（`onLongClick = { viewModel.enterBatchMode(todo.id) }` 保持不变）
- `HomeViewModel.kt`（`enterBatchMode` 已同时完成勾选+多选）
- `SwipeableTodoBox.kt`（左滑手势保持不变）

## 四、手势状态机设计

### 4.1 状态机流程

```
awaitEachGesture {
    down = awaitFirstDown()
    ↓
    发射 Press 事件（水波纹开始）
    启动长按定时器（500ms，协程 job）
    ↓
    ┌─────────────────────────────────────┐
    │ 循环 awaitPointerEvent()            │
    └─────────────────────────────────────┘
        │
        ├─ changedToUp()（手指抬起）
        │   ├─ 定时器未触发（< 500ms）→ 取消定时器
        │   │   └─ 发射 Release → onTap()
        │   └─ 定时器已触发（≥ 500ms）→ 取消定时器
        │       └─ 发射 Release → onLongClick()  ★核心变化
        │
        └─ 移动距离 > touchSlop
            ├─ 定时器未触发 → 取消定时器
            │   └─ 发射 Cancel → break（交给左滑）
            └─ 定时器已触发 → 取消定时器
                └─ 发射 Cancel → break（交给左滑）
}
```

### 4.2 关键参数

| 参数 | 值 | 说明 |
|------|------|------|
| 长按阈值 | 500ms | Compose 默认值，符合"500ms以上"要求 |
| 移动取消阈值 | `viewConfiguration.touchSlop` | 系统默认触控偏移量（约 8dp） |
| `pointerInput` key | `(isBatchMode, onClick, onLongClick, onSelectClick)` | 保持与当前一致 |

### 4.3 与当前实现的差异

| 行为 | 当前（detectTapGestures） | 新实现（awaitEachGesture） |
|------|--------------------------|---------------------------|
| 长按触发时机 | 500ms 到达立即执行 onLongPress | 500ms 到达标记，**手指抬起后**执行 |
| 长按后移动 | 不检测（onLongPress 已触发） | 取消，交给左滑 |
| 短按 | onTap | onTap（不变） |
| 水波纹 | onPress 自动处理 | 手动发射 Press/Release/Cancel |

## 五、数据流

```
手指抬起 + longPressTriggered=true
  → onLongClick() 回调
  → viewModel.enterBatchMode(todo.id)
  → _isBatchMode = true（显示多选页面）
  → _selectedTodoIds = setOf(todoId)（勾选√标记）
```

两个操作（勾选+多选）由 `enterBatchMode` 同步完成，满足"同时执行"要求。

## 六、水波纹与触觉反馈

### 6.1 水波纹（interactionSource）

沿用当前 `interactionSource` 机制，手动管理 PressInteraction 事件：

| 时机 | 动作 |
|------|------|
| `awaitFirstDown()` 后 | `emit(PressInteraction.Press(pressOffset))` |
| 手指抬起 + 执行 onTap/onLongClick | `emit(PressInteraction.Release)` |
| 移动取消 / 协程取消 | `emit(PressInteraction.Cancel)` |

沿用 `try/finally` 兜底策略，确保 Cancel 总是被发射（避免水波纹残留）。

### 6.2 触觉反馈

| 时机 | 类型 |
|------|------|
| 500ms 定时器触发 | `InteractionType.LONG_CLICK`（与当前一致） |

## 七、边界情况与错误处理

| 边界情况 | 处理方式 |
|---------|---------|
| 长按定时器协程被取消（重组） | `longPressTriggered` 保持 false，不触发 onLongClick |
| `pointerInput` key 变化导致手势重启 | 旧协程的 job 自动取消，`longPressTriggered` 重置 |
| 批量模式下长按 | 保持当前语义：`onLongClick` → `toggleSelection` |
| 长按后手指不动 + 抬起 | 正常触发 onLongClick → enterBatchMode |
| 长按后手指微动（< touchSlop） | 不取消，视为不动，抬起仍触发 |
| 快速连续长按多张卡片 | 每张卡片独立手势检测，互不干扰 |

## 八、保留的行为

以下行为**不变**：

1. **批量模式下点击**：`onTap` → `onSelectClick()`（切换选中）
2. **普通模式下点击**：`onTap` → `onClick()`（进入编辑页）
3. **批量模式下长按**：`onLongClick` → `enterBatchMode` 已在批量模式 → 实际走 `toggleSelection`
4. **左滑手势**：SwipeableTodoBox 的 `detectHorizontalDragGestures` 不受影响
5. **LocalContentIndication 水波纹开关**：SwipeableTodoBox 左滑时关闭水波纹的机制不变
