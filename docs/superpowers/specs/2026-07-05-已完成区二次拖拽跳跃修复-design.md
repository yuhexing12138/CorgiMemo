## 1. 背景与问题

### 1.1 用户报告的场景

已知 10 条待办，名称为 1-10，从上到下按序排列。**8, 9, 10 是已完成状态**，其他都是待完成状态。当用户首次将 7 拖拽到已完成区释放是正常的，但**再次在已完成区拖动 7 释放后，7 会跳跃回到待完成区**。

### 1.2 影响

已完成区内的二次拖拽无法正常使用，拖拽释放后被拖项会被错误地移回待完成区。

---

## 2. 根因分析

### 2.1 触发链路

```
用户在已完成区拖动 7（同区域内拖拽）
  ↓
库的 findTargetItem 找 targetItem
  ↓
库跳过 CompletedDivider（divider 不在 reorderableKeys 中）
  ↓
把 divider 后面的项（6）作为 targetItem
  ↓
onMove(8, 6) 被调用，7 跨过 divider 到 pending 区
  ↓
onDragStopped: draggedCurrentIndex=6（pending 区）
  ↓
onReorder(8, 6, 7, false)
  ↓
ViewModel: fromCompleted && toPending → status=0
  ↓
7 跳跃回待完成区
```

### 2.2 库层根因

库的 `findTargetItem`（[ReorderableLazyCollection.kt:593-611](file:///c:/Users/EDY/Desktop/CorgiMemo/Reorderable/reorderable/src/commonMain/kotlin/sh/calvin/reorderable/ReorderableLazyCollection.kt#L593-611)）：

```kotlin
val targetItemFunc = { item: LazyCollectionItemInfo<T> ->
    shouldItemMove(draggingItemRect, targetItemRect)
            && item.key in reorderableKeys  // ← divider 不在这里，被跳过
            && additionalPredicate(item)
}
```

- `reorderableKeys` 只包含 `enabled=true` 的项
- divider 的 `enabled=false`（因为不可拖拽），不在 `reorderableKeys` 中
- 库会跳过 divider，把 divider 后面的项作为 targetItem
- 导致 `onMove(from, to)` 中 from 和 to 跨过 divider

### 2.3 ViewModel 层验证

通过 2 个测试验证 ViewModel 层逻辑正确：

| 测试 | 场景 | 结果 | 结论 |
|------|------|------|------|
| `已完成项在已完成区内拖拽不应改 status` | fromIndex=8, toIndex=9（同区域内） | PASSED | ViewModel 层正确（不改 status） |
| `已完成项被拖到 pending 区应改 status` | fromIndex=8, toIndex=6（跨区） | PASSED | ViewModel 层正确处理跨区 |

**结论**：ViewModel 层逻辑正确，问题在 **UI 层的 onMove**，它允许已完成项跨过 CompletedDivider 被移到 pending 区。

### 2.4 问题本质

- **onMove 无区域边界保护**：onMove 只检查 `isDraggable`，不检查 from 和 to 是否跨过 divider
- **库跳过 divider 找 targetItem**：divider 不可拖拽，库跳过它，把 divider 后面的项作为 targetItem
- **导致误跨区**：用户同区域内拖动时，onMove 让被拖项跨过 divider 到另一个区域
- **ViewModel 误判跨区**：onDragStopped 中 draggedCurrentIndex 落在另一个区域，ViewModel 检测到跨区，改 status

## 3. 设计方案

### 3.1 方案选型

| 方案 | 描述 | 跨区可用性 | 复杂度 |
|------|------|-----------|--------|
| 方案 A | 阻止 `completed → pending` 跨 divider | 单向（pending→completed） | 低 |
| 方案 B | 阻止所有跨 divider 移动 | 无跨区 | 低 |
| **方案 G + 紧邻位置检查（采用）** | onMove 阻止跨 divider，记录 attemptedCrossDivider；onDragStopped 中检测"attemptedCrossDivider + 紧邻位置"触发跨区 | 双向跨区 | 中 |

### 3.2 选型理由

选择方案 G + 紧邻位置检查的核心理由：

1. **双向跨区**：保留 `pending → completed` 和 `completed → pending` 跨区拖拽
2. **修复误跨区**：onMove 阻止跨 divider，避免同区域内拖拽时的误跨区
3. **明确跨区意图**：通过"attemptedCrossDivider + 紧邻位置"双重检查，减少误触发
4. **用户体验好**：被拖项不会跨 divider（视觉上停留在原区域），释放时如果检测到跨区意图，触发跨区

### 3.3 算法设计

#### 3.3.1 新增状态变量

```kotlin
/** 是否尝试跨 divider（onMove 阻止跨 divider 时记录） */
var attemptedCrossDivider by remember { mutableStateOf(false) }
```

#### 3.3.2 onMove 修改

```kotlin
onMove = { from, to ->
    // ① 原 isDraggable 检查
    val fromItem = displayItems.getOrNull(from.index)
    val toItem = displayItems.getOrNull(to.index)
    if (fromItem == null || toItem == null) return@rememberReorderableLazyListState
    if (!isDraggable(fromItem) || !isDraggable(toItem)) {
        return@rememberReorderableLazyListState
    }

    // ② [新增] 阻止跨 divider 移动，但记录尝试
    if (dividerIndex >= 0) {
        val minIdx = minOf(from.index, to.index)
        val maxIdx = maxOf(from.index, to.index)
        if (minIdx < dividerIndex && maxIdx > dividerIndex) {
            attemptedCrossDivider = true  // 记录尝试跨 divider
            return@rememberReorderableLazyListState  // 阻止
        }
    }

    // ③ 原重排逻辑
    val newDisplay = displayItems.toMutableList()
    val draggedItem = newDisplay.removeAt(from.index)
    newDisplay.add(to.index, draggedItem)
    displayItems = newDisplay

    // ④ 原触觉反馈
    // ...
}
```

#### 3.3.3 onDragStopped 修改

```kotlin
onDragStopped = {
    val draggedCurrentIndex = displayItems.indexOfFirst {
        key(it) == key(item)
    }

    // [新增] 跨 divider 检测：如果尝试跨 divider 且被拖项在 divider 紧邻位置，触发跨区
    var crossDividerTriggered = false
    if (attemptedCrossDivider && dividerIndex >= 0 && draggedOriginalIndex >= 0 &&
        draggedCurrentIndex >= 0
    ) {
        val originalInPending = draggedOriginalIndex < dividerIndex
        // 双重检查：尝试跨 divider + 被拖项在 divider 紧邻位置
        val isAdjacentToDivider = draggedCurrentIndex == dividerIndex - 1 ||
            draggedCurrentIndex == dividerIndex + 1
        if (isAdjacentToDivider) {
            // 跨区目标位置：divider 另一侧紧邻位置
            val effectiveToIndex = if (originalInPending) dividerIndex + 1 else dividerIndex - 1

            crossedPinnedZone = ReorderAlgorithms.checkPinnedZoneCrossed(
                displayItems = displayItems,
                isPinned = isPinned,
                dividerKind = dividerKind,
                draggedOriginalIsPinned = draggedOriginalIsPinned,
                draggedCurrentIndex = effectiveToIndex
            )
            onReorder(
                draggedOriginalIndex,
                effectiveToIndex,
                dividerIndex,
                crossedPinnedZone
            )
            HapticFeedbackManager.performHapticFeedback(
                context = context,
                type = InteractionType.CONFIRM,
                enabled = true
            )
            crossDividerTriggered = true
        }
        // 如果 attemptedCrossDivider=true 但不在紧邻位置，回退到原逻辑
    }

    // 原逻辑：同区域内拖拽（仅在未触发跨区时执行）
    if (!crossDividerTriggered && draggedOriginalIndex >= 0 &&
        draggedOriginalIndex != draggedCurrentIndex &&
        draggedCurrentIndex >= 0
    ) {
        crossedPinnedZone = ReorderAlgorithms.checkPinnedZoneCrossed(...)
        onReorder(draggedOriginalIndex, draggedCurrentIndex, dividerIndex, crossedPinnedZone)
        // ...
    }

    // 重置状态
    isDragActive = false
    draggedOriginalIndex = -1
    crossedPinnedZone = false
    attemptedCrossDivider = false
}
```

**注意**：使用 `crossDividerTriggered` 变量避免 lambda 中的 return，保持代码结构清晰。

### 3.4 关键不变式

1. **onMove 阻止跨 divider**：from 和 to 跨 divider 时，return，不执行重排
2. **attemptedCrossDivider 记录意图**：onMove 阻止跨 divider 时，设置 attemptedCrossDivider=true
3. **onDragStopped 双重检查**：attemptedCrossDivider=true 且被拖项在 divider 紧邻位置时，触发跨区
4. **双向跨区**：pending→completed（toIndex=dividerIndex+1）和 completed→pending（toIndex=dividerIndex-1）都可用
5. **同区域内拖拽不跨区**：onMove 不跨 divider，attemptedCrossDivider 保持 false

### 3.5 用户场景验证

| 场景 | onMove | attemptedCrossDivider | draggedCurrentIndex | onDragStopped | 结果 |
|------|--------|------------------------|---------------------|---------------|------|
| 同区域内拖拽（7 拖到 8/9 之间） | 不跨 divider | false | 9（completed 区） | 不触发跨区，调用原逻辑 | 7 留在 completed 区 ✓ |
| 首次 pending→completed 跨区（7 拖向下越过 divider） | 尝试跨 divider，阻止 | true | 6（pending 区，紧邻 divider） | 触发跨区，toIndex=divider+1 | 7 移到 completed 区 ✓ |
| 二次 completed→pending 跨区（7 拖向上越过 divider） | 尝试跨 divider，阻止 | true | 8（completed 区，紧邻 divider） | 触发跨区，toIndex=divider-1 | 7 移回 pending 区 ✓ |
| 误触发（用户拖到 divider 附近但释放时拖回原区域） | 不跨 divider | false | 原区域 | 不触发跨区 | 7 留在原区域 ✓ |

## 6. 风险与缓解

### 6.1 误触发风险

**风险**：用户拖动被拖项到 divider 附近，onMove 尝试跨 divider，触发 attemptedCrossDivider=true。如果用户释放时被拖项在 divider 紧邻位置，会触发跨区。

**缓解**：通过"attemptedCrossDivider + 紧邻位置"双重检查，减少误触发。只有当用户明确拖动被拖项到 divider 紧邻位置且尝试跨 divider 时，才触发跨区。

### 6.2 边界情况

**风险**：dividerIndex=-1（无已完成区）时，attemptedCrossDivider 不应触发。

**缓解**：onMove 中检查 `dividerIndex >= 0`，无已完成区时不执行跨 divider 检查。

### 6.3 滚动场景

**风险**：用户拖动被拖项到列表边缘触发滚动，可能导致 onMove 行为异常。

**缓解**：onMove 的跨 divider 检查不依赖滚动状态，只检查 from 和 to 的 index。
