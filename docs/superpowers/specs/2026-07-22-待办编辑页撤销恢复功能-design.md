# 待办编辑页撤销/恢复功能设计

> 日期：2026-07-22
> 状态：已审批
> 范围：TodoEditScreen 撤销/恢复功能

## 1. 背景与动机

当前待办编辑页（TodoEditScreen）的 ViewModel 中已存在双栈撤销/恢复架构（纯文本栈 + 扩展栈），但 **UI 层未接入**。`TodoGroupContainer`（CheckboxEditText.kt）已预留了 `canUndo/canRedo/onUndoClick/onRedoClick` 参数接口和 `↩ 撤销` / `↪ 恢复` 文字按钮 UI，但 `TodoEditScreen` 调用时未传入参数。

现有双栈架构存在以下问题：
- **覆盖范围窄**：仅支持文本快照和内容块操作，不支持元数据（分类、日期、位置等）的撤销
- **架构复杂**：两套栈 + 批量合并 + 持久化，维护成本高
- **UI 未激活**：全部后端逻辑无法被用户感知

本设计将 **移除双栈，替换为统一全量快照栈**，覆盖全部可编辑字段（排除背景色），并激活 UI 层连接。

## 2. 需求确认

| 维度 | 决定 |
|------|------|
| **覆盖范围** | 全部字段（排除背景色）：文本、分类、优先级、提醒、日期、位置、附件、关联 |
| **UI 位置** | 使用 TodoGroupContainer 已预留的 `↩ 撤销` / `↪ 恢复` 按钮 |
| **快照粒度** | 全量快照（每次变更记录完整编辑状态） |
| **保存后行为** | 保留撤销栈（保存后仍可撤销到保存前状态） |
| **持久化** | 仅会话内（退出编辑页后清空，不持久化到磁盘） |

## 3. 方案选型

### 3.1 方案对比

| 方案 | 描述 | 优点 | 缺点 |
|------|------|------|------|
| **A：统一全量快照栈** ⭐ | 移除双栈，替换为单一 `EditSnapshot` 全量快照栈 | 架构简洁统一，撤销/恢复逻辑一致，与需求完美匹配 | 需定义 `EditSnapshot` 数据类并序列化，需移除现有双栈代码 |
| B：扩展双栈 | 保留现有双栈，新增元数据栈 | 改动较小 | 三栈协调复杂，撤销顺序不一致，维护负担大 |
| C：Memento + StateFlow | 利用 StateFlow 特性，每次状态变更自动记录 Memento | 与 Compose 响应式架构契合 | 需重构 ViewModel 为单一状态类，改动范围极大 |

### 3.2 选定方案

**方案 A：统一全量快照栈**。理由：
1. 与需求完全匹配（全字段覆盖）
2. 架构简洁——一个栈管全部，不会出现"文本已恢复但分类没恢复"的问题
3. UI 已预留接口，只需传入参数即可激活
4. 灵感编辑页未来可复用相同模式

## 4. 数据模型

### 4.1 EditSnapshot 数据类

全量快照，记录编辑页某一时刻的所有可编辑字段状态（排除背景色）：

```kotlin
/**
 * 编辑状态全量快照，用于撤销/恢复功能
 * 记录编辑页某一时刻的所有可编辑字段状态（排除背景色）
 */
data class EditSnapshot(
    // 文本内容
    val title: String,
    val content: String,
    val contentFormat: String,
    val subTasks: List<SubTask>,
    val contentBlocks: List<ContentBlock>,

    // 单值元数据
    val startDate: Long?,
    val dueDate: Long?,

    // 位置
    val geofenceLat: Double,
    val geofenceLng: Double,
    val geofenceRadius: Float,
    val geofenceType: Int,
    val geofenceEnabled: Boolean,
    val geofenceAddress: String,

    // 附件
    val imagePaths: List<String>,
    val voiceNotePath: String?,
    val voiceDuration: Int?,
    val lineAttachmentsSnapshot: String?,

    // 多分组字段
    val groupCategoryIds: Map<Int, Long>,
    val groupPriorities: Map<Int, Int>,
    val groupReminders: Map<Int, Long?>,
    val groupRepeatTypes: Map<Int, Int>,
    val groupRelations: Map<Int, List<CardRelation>>,
)
```

**设计决策**：
- **排除背景色**：用户明确不需要背景色的撤销/恢复
- **排除 `groupSaveStates`**：分组保存状态是 UI 状态，不属于"编辑内容"
- **多分组字段用 Map**：与 ViewModel 中现有的 Map 结构一致，避免序列化转换

### 4.2 UndoRedoStack 栈管理器

```kotlin
/**
 * 撤销/恢复栈管理器
 * 维护两个 ArrayDeque，深度限制 MAX_SNAPSHOT_DEPTH = 50
 */
class UndoRedoStack(
    private val maxDepth: Int = MAX_SNAPSHOT_DEPTH
) {
    private val undoStack = ArrayDeque<EditSnapshot>(maxDepth)
    private val redoStack = ArrayDeque<EditSnapshot>(maxDepth)

    val canUndo: Boolean get() = undoStack.isNotEmpty()
    val canRedo: Boolean get() = redoStack.isNotEmpty()

    /** 推入快照到撤销栈，清空恢复栈 */
    fun push(snapshot: EditSnapshot) {
        if (undoStack.size >= maxDepth) {
            undoStack.removeFirst()
        }
        undoStack.addLast(snapshot)
        redoStack.clear()
    }

    /** 撤销：当前状态入恢复栈，弹出撤销栈顶 */
    fun undo(current: EditSnapshot): EditSnapshot? {
        if (undoStack.isEmpty()) return null
        redoStack.addLast(current)
        return undoStack.removeLast()
    }

    /** 恢复：当前状态入撤销栈，弹出恢复栈顶 */
    fun redo(current: EditSnapshot): EditSnapshot? {
        if (redoStack.isEmpty()) return null
        undoStack.addLast(current)
        return redoStack.removeLast()
    }

    /** 清空所有栈 */
    fun clear() {
        undoStack.clear()
        redoStack.clear()
    }

    companion object {
        const val MAX_SNAPSHOT_DEPTH = 50
    }
}
```

**关键设计决策**：

| 决策 | 选择 | 原因 |
|------|------|------|
| 快照深度 | 50 层 | 与现有 MAX_UNDO_DEPTH 一致 |
| 恢复栈清空时机 | 推入新快照时 | 标准撤销/恢复语义：新操作后无法恢复 |
| 栈溢出处理 | 移除栈底（FIFO） | 保留最近的操作历史 |

## 5. 快照采集与恢复

### 5.1 采集时机

核心原则：**在字段值变更之前**推入当前状态快照。

| 触发场景 | 采集时机 | 说明 |
|---------|---------|------|
| 文本编辑 | `todoLines` 变更的 `LaunchedEffect` 触发前 | **防抖**：500ms 内连续按键只记录一次快照 |
| 分类变更 | `setCategory()` 调用前 | 即时采集 |
| 优先级变更 | `setPriority()` 调用前 | 即时采集 |
| 提醒时间变更 | `setReminder()` 调用前 | 即时采集 |
| 重复类型变更 | `setRepeatType()` 调用前 | 即时采集 |
| 日期变更 | `setStartDate()`/`setDueDate()` 调用前 | 即时采集 |
| 位置变更 | `setGeofence*()` 系列调用前 | 即时采集 |
| 图片附件变更 | `setImagePaths()` 调用前 | 即时采集 |
| 语音备注变更 | `setVoiceNotePath()` 调用前 | 即时采集 |
| 关联变更 | `setRelations()` 调用前 | 即时采集 |
| 内容块变更 | `setContentBlocks()` 调用前 | 即时采集 |

### 5.2 文本编辑防抖

文本编辑是高频操作（每次按键都触发），需要防抖以避免快照爆炸：

```
首次按键 → 立即推入当前快照（变更前状态）→ 启动 500ms 冷却
后续按键（冷却期内）→ 不推入快照
冷却期结束 → 下一次按键重新触发快照推入
```

**关键**：快照必须在第一次变更时**立即推入**（记录变更前状态），而非延迟推入。防抖的作用是避免连续按键产生大量中间快照。

```kotlin
private var debounceJob: Job? = null
private var isDebouncing = false

fun pushSnapshotDebounced() {
    if (isDebouncing) return  // 冷却期内，不推入
    isDebouncing = true
    pushSnapshot()  // 立即推入当前快照
    debounceJob?.cancel()
    debounceJob = viewModelScope.launch {
        delay(500)
        isDebouncing = false  // 冷却期结束
    }
}

fun pushSnapshot() {
    undoRedoStack.push(currentSnapshot)
    _canUndo.value = undoRedoStack.canUndo
    _canRedo.value = undoRedoStack.canRedo
}
```

### 5.3 恢复机制

```kotlin
/**
 * 从快照恢复编辑状态
 * 将快照中的所有字段值写回对应的 StateFlow
 */
fun restoreFromSnapshot(snapshot: EditSnapshot) {
    _title.value = snapshot.title
    _content.value = snapshot.content
    _contentFormat.value = snapshot.contentFormat
    // 元数据
    _startDate.value = snapshot.startDate
    _dueDate.value = snapshot.dueDate
    // 位置
    _geofenceLat.value = snapshot.geofenceLat
    _geofenceLng.value = snapshot.geofenceLng
    _geofenceRadius.value = snapshot.geofenceRadius
    _geofenceType.value = snapshot.geofenceType
    _geofenceEnabled.value = snapshot.geofenceEnabled
    _geofenceAddress.value = snapshot.geofenceAddress
    // 附件
    _imagePaths.value = snapshot.imagePaths
    _voiceNotePath.value = snapshot.voiceNotePath
    _voiceDuration.value = snapshot.voiceDuration
    _lineAttachmentsSnapshot.value = snapshot.lineAttachmentsSnapshot
    // 多分组字段
    _groupCategoryIds.value = snapshot.groupCategoryIds
    _groupPriorities.value = snapshot.groupPriorities
    _groupReminders.value = snapshot.groupReminders
    _groupRepeatTypes.value = snapshot.groupRepeatTypes
    _groupRelations.value = snapshot.groupRelations

    // 通知 UI 层重建 todoLines
    _restoreEvent.value = snapshot
}
```

### 5.4 防循环机制

恢复快照会修改 StateFlow → 触发 `LaunchedEffect(todoLines)` → 又推入新快照 → 无限循环。

**解决方案**：引入 `_isRestoring` 标志：

```kotlin
private val _isRestoring = MutableStateFlow(false)

// LaunchedEffect 中：
if (!_isRestoring.value) {
    pushSnapshotDebounced()
}

// 恢复时：
_isRestoring.value = true
restoreFromSnapshot(snapshot)
_isRestoring.value = false
```

### 5.5 UI 层恢复事件

恢复快照后，需要通知 UI 层重建 `todoLines`（因为 `todoLines` 是本地 Compose 状态）：

```kotlin
// ViewModel 中
private val _restoreEvent = MutableStateFlow<EditSnapshot?>(null)
val restoreEvent: StateFlow<EditSnapshot?> = _restoreEvent

// TodoEditScreen 中
LaunchedEffect(restoreEvent) {
    restoreEvent?.let { snapshot ->
        todoLines = buildTodoLinesFromSnapshot(snapshot)
    }
}
```

### 6.4 undo/redo 新实现

```kotlin
fun undo() {
    val current = currentSnapshot
    val previous = undoRedoStack.undo(current) ?: return
    _isRestoring.value = true
    restoreFromSnapshot(previous)
    _isRestoring.value = false
    _canUndo.value = undoRedoStack.canUndo
    _canRedo.value = undoRedoStack.canRedo
}

fun redo() {
    val current = currentSnapshot
    val next = undoRedoStack.redo(current) ?: return
    _isRestoring.value = true
    restoreFromSnapshot(next)
    _isRestoring.value = false
    _canUndo.value = undoRedoStack.canUndo
    _canRedo.value = undoRedoStack.canRedo
}
```

### 7.3 buildTodoLinesFromSnapshot 辅助函数

将快照中的数据转换为 `List<TodoLine>`。实际实现需要从 `lineAttachmentsSnapshot`（JSON 字符串）解析各行的附件数据，这里展示核心逻辑：

```kotlin
fun buildTodoLinesFromSnapshot(snapshot: EditSnapshot): List<TodoLine> {
    // 从 lineAttachmentsSnapshot 解析行级附件映射
    val attachmentMap = parseLineAttachmentsSnapshot(snapshot.lineAttachmentsSnapshot)

    // 首行作为标题行
    val titleLine = TodoLine(
        text = snapshot.title,
        isChecked = false,
        isTitle = true,
        imagePaths = attachmentMap[0]?.imagePaths ?: snapshot.imagePaths,
        voiceAttachments = attachmentMap[0]?.voiceAttachments
            ?: snapshot.voiceNotePath?.let { listOf(it) } ?: emptyList(),
    )
    // 子任务行
    val subTaskLines = snapshot.subTasks.mapIndexed { index, subTask ->
        val lineIndex = index + 1  // 标题行占 index 0
        TodoLine(
            text = subTask.content,
            isChecked = subTask.isCompleted,
            isTitle = false,
            imagePaths = attachmentMap[lineIndex]?.imagePaths ?: emptyList(),
            voiceAttachments = attachmentMap[lineIndex]?.voiceAttachments ?: emptyList(),
        )
    }
    return listOf(titleLine) + subTaskLines
}
```

## 8. 多分组场景

当前待办编辑页支持多分组（子 todo），每个分组有独立的分类/优先级/提醒等。快照中包含所有分组的 Map 状态，撤销/恢复时所有分组的状态一起恢复。

**语义**：撤销是全局的，不是按分组独立撤销。这与用户对"撤销"的直觉一致——撤销刚才的操作，包括它对所有分组的影响。

## 10. 测试要点

| 场景 | 验证内容 |
|------|---------|
| 基本文本编辑撤销 | 输入文字 → 撤销 → 文字恢复到之前状态 |
| 基本文本编辑恢复 | 撤销后 → 恢复 → 文字回到撤销前状态 |
| 元数据撤销 | 修改分类/优先级/日期 → 撤销 → 元数据恢复 |
| 位置撤销 | 修改地理围栏 → 撤销 → 位置恢复 |
| 附件撤销 | 添加图片/语音 → 撤销 → 附件恢复 |
| 关联撤销 | 添加关联 → 撤销 → 关联恢复 |
| 保存后撤销 | 保存 → 撤销 → 状态恢复到保存前 |
| 连续撤销 | 多次操作 → 连续撤销 → 逐步恢复 |
| 新操作清空恢复栈 | 撤销后 → 新操作 → 恢复按钮禁用 |
| 深度限制 | 连续操作 > 50 次 → 最早的快照被丢弃 |
| 防抖验证 | 快速输入多个字符 → 只生成一个快照 |
| 防循环验证 | 撤销/恢复不会导致无限循环 |
| 退出编辑页 | 栈清空，重新进入时无历史 |

## 11. 后续优化建议

1. **灵感编辑页复用**：将 `EditSnapshot` / `UndoRedoStack` 模式应用到 InspirationEditScreen，替换其当前基于 `richTextState.history` 的方案
2. **物理键盘快捷键**：支持 Ctrl+Z / Ctrl+Shift+Z
3. **撤销提示 Toast**：撤销时显示轻量提示，告知用户撤销了什么操作
4. **快照内存优化**：如果内存成为问题，可考虑增量压缩（如只存储差异部分）
